# 기존 콘텐츠 정리 인벤토리 v1

이 문서는 현재 `TIL`에 쌓여 있는 문서를 어떤 기준으로 정리할지 결정하기 위한 운영 문서다.

`TIL`은 빠르게 적재하는 초안 저장소로 유지하고, `wiki`는 읽기 좋은 정리본만 올리는 것을 전제로 한다.

## 1. 현재 상태 요약

- 전체 Markdown 문서 수: `47`
- 루트에 바로 놓인 문서 수: `0`
- 폴더 내부 문서 수: `47`
- `wiki` 현재 상태: `Home.md`와 첫 대표 문서 1개 존재

현재 구조에서 먼저 보이는 문제는 다음과 같다.

- 루트 문서 1차 재배치는 끝났지만, 중복 문서는 아직 남아 있다.
- 같은 주제를 `주제 폴더`, `Tech Blog`, 세부 문서로 중복 보관한 흔적이 있다.
- 문서명 규칙이 혼재되어 있다.
- `wiki`로 승격할 만한 글이 이미 있는데도, 아직 대표 문서가 없다.

## 2. 운영 분류 기준

문서를 정리할 때는 아래 네 가지 상태로 판단한다.

- `TIL 유지`: 초안, 학습 흔적, 러프한 정리. 현재 저장소에 그대로 둔다.
- `재배치 필요`: 내용은 유지하되 루트에서 주제 폴더로 옮긴다.
- `병합 후보`: 같은 주제를 다루는 문서가 둘 이상이라 대표 문서를 정해야 한다.
- `위키 승격 후보`: 구조와 내용이 이미 충분하거나, 조금만 다듬으면 `wiki` 대표 문서가 될 수 있다.

## 2.1 위키 문체 기준

`wiki` 문체는 `블로그 글`과 `백과사전식 정리`의 중간쯤으로 맞춘다.

- 처음 보는 사람도 흐름을 따라 이해할 수 있게 쓴다.
- 너무 건조한 요약문보다 맥락과 이유를 함께 적는다.
- 그렇다고 개인 감상문처럼 흘러가지 않고, 구조와 제목 체계는 분명하게 유지한다.
- 한 문서 안에서 `왜 필요한가`, `어떻게 동작하는가`, `어디서 자주 막히는가`가 보여야 한다.
- 문체는 설명형이되 과한 홍보 톤이나 과장된 표현은 피한다.

## 6. 1차 실행 배치: Spring Security

- 문서 수가 많다.
