# TIL 저장소 전용 규칙

이 저장소는 공부한 내용을 어떻게 정리하고 축적할지에 대한 규칙을 담는다.

## 역할

공부 중 다룬 내용은 두 곳에 나눠 기록한다.

- `sessions/`: 그날 대화의 원본 — 다룬 질문, 결론, 근거를 그대로 남긴다.
- 각 도메인 폴더(`Database/`, `JPA/`, `네트워크/` 등): 그중 확정된 개념/결론만 승격해 정리한다. 새 도메인이 필요하면 루트에 폴더를 새로 만든다.

근거와 출처를 바탕으로 정리하고, 추측이 필요하면 "모름" 또는 검증 방법을 제시한다 — 추측을 단정한 사실처럼 적지 않는다.

## 기록 규칙

1. 세션마다 `sessions/YYYY-MM-DD_<slug>.md`를 만든다 — 그날 다룬 질문, 결론, 근거, 검증 상태를 남긴다(세션 단위 체크리스트로만 추적 — 도메인 노트에는 별도 검증 상태 필드를 두지 않는다).
2. 세션 중 확정된 개념/결론은 해당 도메인 폴더로 승격한다. 해당 도메인 폴더가 없으면 새로 만든다.
   - 파일명은 한글 가능. 띄어쓰기 대신 하이픈(`-`)으로 연결 (예: `런타임-데이터-영역.md`).
3. 모든 주장에는 출처를 단다 (공식 문서/소스코드/실측/책 등 1차 출처 우선. 블로그·커뮤니티 같은 2차 출처만 있는 경우 본문에 그 사실을 명시한다).
4. 출처 인용은 본문에 텍스트로 풀어쓰지 않고(예: "(JVMS §2.5.5)") 마크다운 링크로 직접 연결한다 (`[JVMS §2.5.5](URL)`). 여러 섹션/항목을 인용할 때는 한 줄에 나열하지 않고 하위 목록으로 분리한다.
5. 기존 노트와 상충하는 정보가 나오면 본문을 갱신하고 `최종 수정일`을 갱신한다.
6. 노트를 새로 만들거나 크게 갱신하면 `INDEX.md`에 링크를 추가/갱신하고, 해당 도메인 폴더에 `README.md`가 있다면 그 목차에도 추가한다.
7. 템플릿은 `_TEMPLATE.md`(도메인 노트용), `sessions/_TEMPLATE.md`(세션 노트용)를 따른다.
8. `Kafka/`, `Elasticsearch/`, `Spring Webflux/`는 별도 저장소를 가리키는 git submodule이다 — 이 폴더들 내부는 직접 수정하지 않는다. Webflux 관련 학습 노트는 `Spring WebFlux 노트/`에 별도로 쌓는다.

## 톤

- 결론 먼저, 근거 뒤. 장황한 서론 금지.
- 불확실하면 추측 말고 "모름" 또는 검증 방법 제시.
- 선택지 나열만 하지 말고 추천안 하나 제시 + 이유.
- 설계 질문에는 트레이드오프 표(일관성/가용성/지연/비용/운영부담)로 답한다. 규모 가정(QPS, 데이터 크기)을 명시하고, 가정이 다르면 답이 달라짐을 표시한다.
- 코드는 프로덕션 기준(에러 처리, 엣지 케이스, 동시성)을 따른다. 보안 민감 코드(인증/입력검증/쿼리)는 절대 축약·생략하지 않는다. SQL은 인덱스/실행계획 영향을 언급한다. 주석은 "왜"만 남기고 "무엇"은 코드로 표현한다.
- 칭찬·사과·빈말은 쓰지 않는다. 이미 정한 결정은 재논쟁하지 않는다. 검증 안 된 라이브러리 API는 단정하지 말고 최신 문서 확인을 권장한다.

자세한 문서 작성 컨벤션(백틱 사용 범위, 코드 예시 언어 등)은 `CONTRIBUTING.md`를 따른다.
