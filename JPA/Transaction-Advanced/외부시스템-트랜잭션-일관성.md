# 외부 시스템과 트랜잭션 일관성

> 태그: `#jpa` `#transaction` `#outbox-pattern` `#saga` `#idempotency`<br>
> 작성일: 2026-06-24<br>
> 최종 수정일: 2026-06-24

## 정의

DB 트랜잭션은 외부 API·메시지 발행까지 원자적으로 묶지 못하므로, 유실 허용 여부에 따라 Outbox Pattern, `@TransactionalEventListener(AFTER_COMMIT)`, Saga, Idempotency Key 중 적합한 전략을 선택해 일관성을 확보한다.

## 특징 / 상세

### 1. DB 트랜잭션은 외부 API까지 원자적으로 묶지 못한다

다음 코드는 흔하지만 위험하다.

```java
@Transactional
public void approvePayment(Long orderId) {
    Order order = orderRepository.getById(orderId);
    paymentClient.approve(order.getPaymentKey());
    order.markPaid();
}
```

가능한 실패 시나리오:

- 결제 API는 성공했지만 DB 커밋이 실패한다.
- DB 상태는 변경됐지만 외부 API 호출이 timeout으로 실패한다.
- timeout이 났지만 외부 시스템에서는 실제로 성공했을 수 있다.

DB와 외부 시스템은 하나의 로컬 트랜잭션으로 묶이지 않는다. 따라서 즉시 일관성을 강제로 만들기보다, 실패를 추적하고 복구할 수 있는 구조가 필요하다.

### 2. Outbox Pattern

Outbox Pattern은 DB 변경과 이벤트 저장을 하나의 트랜잭션에 넣는다.

```java
@Transactional
public void placeOrder(OrderCommand command) {
    Order order = Order.create(command);
    orderRepository.save(order);

    outboxRepository.save(OutboxEvent.of(
        "ORDER_PLACED",
        order.getId(),
        OrderPlacedPayload.from(order)
    ));
}
```

이후 별도 publisher가 outbox 테이블을 읽어 Kafka, RabbitMQ, 외부 API 등으로 발행한다.

장점:

- 주문 저장과 이벤트 저장은 같은 DB 트랜잭션으로 원자성을 가진다.
- 애플리케이션이 이벤트 발행 직전에 죽어도 outbox row가 남는다.
- 발행 실패를 재시도하거나 수동 보정할 수 있다.

주의할 점:

- 이벤트 발행은 적어도 한 번(at-least-once) 발생할 수 있다.
- 소비자는 멱등적으로 설계해야 한다.
- outbox 테이블 청소, 재시도 횟수, 실패 상태 관리가 필요하다.

### 3. Transactional Event afterCommit

Spring에서는 트랜잭션 커밋 이후 이벤트를 처리할 수 있다.

```java
@Transactional
public void placeOrder(OrderCommand command) {
    Order order = Order.create(command);
    orderRepository.save(order);
    applicationEventPublisher.publishEvent(new OrderPlacedEvent(order.getId()));
}

@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handle(OrderPlacedEvent event) {
    notificationClient.send(event.orderId());
}
```

이 방식은 "커밋된 뒤에만 후속 작업을 실행한다"는 점에서 유용하다. 하지만 프로세스가 커밋 직후 이벤트 처리 전에 종료되면 이벤트가 유실될 수 있다. 유실이 치명적인 이벤트라면 outbox가 더 안전하다.

추가로 주의할 점:

- 이벤트 핸들러가 실패해도 원본 트랜잭션은 이미 커밋된 뒤다.
- 핸들러 재시도, 실패 로그, 수동 보정 경로가 필요하다.
- `@Async`와 함께 쓰면 트랜잭션 컨텍스트는 thread-local로 전파되지 않는다.
- 외부 메시지 브로커 발행은 outbox 없이 처리하면 커밋과 발행 사이의 유실 구간이 남는다.

### 4. Saga와 보상 트랜잭션

여러 서비스 또는 외부 시스템을 거치는 작업은 하나의 ACID 트랜잭션으로 묶기 어렵다. 이때는 각 단계를 로컬 트랜잭션으로 처리하고, 실패 시 보상 작업을 실행한다.

```text
주문 생성
→ 결제 승인
→ 재고 확정
→ 배송 요청
```

배송 요청에서 실패하면 다음과 같은 보상 흐름이 필요할 수 있다.

```text
배송 요청 실패
→ 재고 확정 취소
→ 결제 취소
→ 주문 실패 처리
```

Saga는 "모두 동시에 커밋"이 아니라 "각 단계의 성공과 실패를 상태로 기록하고 복구한다"는 접근이다.

### 5. Idempotency Key

외부 연동에서는 같은 요청이 여러 번 들어오는 것을 기본으로 가정해야 한다.

```java
@Entity
@Table(
    name = "payment_request",
    uniqueConstraints = @UniqueConstraint(
        name = "uk_payment_request_idempotency_key",
        columnNames = "idempotency_key"
    )
)
public class PaymentRequest {
    private String idempotencyKey;
}
```

멱등 키가 있으면 클라이언트 재시도, 서버 재시도, 메시지 중복 소비가 발생해도 같은 비즈니스 효과가 한 번만 생기도록 제어할 수 있다.

### 6. 전략 선택

| 요구 | 권장 전략 |
| --- | --- |
| 커밋 후 알림처럼 유실되어도 치명적이지 않음 | `@TransactionalEventListener(AFTER_COMMIT)` |
| 이벤트 유실이 치명적임 | Outbox Pattern |
| 여러 서비스 상태 전이가 필요함 | Saga |
| 중복 요청 방지가 필요함 | Idempotency Key + unique constraint |
| 외부 API 결과 확인이 필요함 | 상태 machine + 재조회/reconcile 작업 |

### 7. 캐시 무효화도 트랜잭션 이후를 기준으로 본다

DB update와 캐시 삭제/갱신을 같은 코드 흐름에 둬도 하나의 원자적 작업이 되지는 않는다.

```java
@Transactional
public void changeProductName(Long productId, String name) {
    Product product = productRepository.getById(productId);
    product.changeName(name);

    cache.evict("product:" + productId);
}
```

이 코드에서 캐시는 삭제됐지만 DB 트랜잭션은 이후 rollback될 수 있다. 반대로 DB는 commit됐지만 캐시 삭제가 실패할 수도 있다.

실무에서는 다음 기준으로 정리한다.

- 캐시 삭제는 가능하면 commit 이후에 실행한다.
- stale read 허용 시간을 정한다.
- 캐시 갱신 실패를 재시도하거나 TTL로 회복되게 한다.
- 강한 정합성이 필요한 데이터는 캐시 의존도를 낮춘다.
