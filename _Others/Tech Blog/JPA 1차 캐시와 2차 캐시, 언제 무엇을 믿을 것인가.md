# JPA 1차 캐시와 2차 캐시, 언제 무엇을 믿을 것인가

> 연결된 정리본:
> - [조회와 캐시 전략](../../../TIL.wiki/jpa-query-and-cache-strategies.md)


> “JPA를 쓰기 시작한 뒤로, 내가 가장 헷갈렸던 건 ‘도대체 이 데이터는 어디에서 온 건가?’였다.”

## 0. 이 글을 쓰게 된 계기

JPA/Hibernate를 쓰기 시작했을 때, 나는 `findById()`나 `getReference()` 같은 메서드들이 전부 **DB를 직접 때린다고** 생각했다.

> “이 메서드를 호출하면 쿼리가 나가고, 엔티티가 하나씩 만들어지겠지.”

근데 디버깅을 하다 보면, **같은 트랜잭션 안에서 두 번, 세 번 조회하는데도 SQL 로그가 안 찍히는** 걸 보고 혼란이 왔다.

- “분명 `findById()`를 두 번 불렀는데, 쿼리는 한 번만 나가네?”  
- “트랜잭션을 나눴더니 이번엔 또 매번 쿼리가 나가네?”  
- “Redis 캐시도 쓰고 있는데, 그럼 JPA 캐시랑은 무슨 관계지?”

이때부터 자연스럽게 **1차 캐시와 2차 캐시**라는 개념을 다시 보게 됐다.  
정리해 보니, 이 둘은 생각보다 다른 역할과 범위를 가지고 있었다.

이 글은,

- JPA에서 말하는 1차 캐시와 2차 캐시가 각각 **무슨 역할을 하는지**  
- 트랜잭션/엔티티 생명주기와 어떻게 엮여 있는지  
- Redis 같은 애플리케이션 캐시와는 어떤 관계/차이가 있는지

를 한 번에 정리해 보려는 시도다.

---

## 1. 왜 1차 캐시와 2차 캐시를 이해해야 할까?

실무에서 1차/2차 캐시를 제대로 이해하지 못하면 이런 문제가 생긴다.

1. **“쿼리가 언제 나가는지”를 예측하지 못한다**
   - 같은 코드를 환경/설정에 따라 다르게 해석하게 된다.  
   - “로컬에서는 쿼리가 한 번인데, 운영에서는 왜 이렇게 많이 나가지?” 같은 불안이 생긴다.

2. **정합성과 성능 사이에서 잘못된 가정을 하게 된다**
   - 1차 캐시에 기대야 하는 상황에서 Redis 캐시를 덕지덕지 붙인다든지  
   - 2차 캐시에만 의존하다가, 예상치 못한 정합성 이슈를 맞는다.

3. **엔티티 생명주기와 트랜잭션 경계를 헷갈린다**
   - “왜 여기서는 변경 감지가 되는데, 저기서는 안 되지?”  
   - “왜 어떤 곳에서는 `flush()`가 필요하고, 어떤 곳에서는 필요 없지?”

내가 느낀 포인트는 딱 하나였다.

> **“1차 캐시는 트랜잭션 단위의 ‘엔티티 뷰’를 제공하고,  
>  2차 캐시는 애플리케이션 전체에서 재사용 가능한 ‘엔티티 스냅샷’을 제공한다.”**

이 관점을 기준으로 둘을 비교해 보면 훨씬 이해하기 쉬워졌다.

---

## 2. 큰 그림: JPA 캐시 구조를 한 번에 훑어보기

먼저 용어부터 정리해 보자.

- **1차 캐시 (First-Level Cache)**
  - JPA `EntityManager`(Hibernate에서는 `Session`)가 관리하는 **영속성 컨텍스트 내부의 캐시**  
  - 트랜잭션/요청 단위로 생겼다가 사라지는, **가장 가까운 캐시**
- **2차 캐시 (Second-Level Cache)**
  - `EntityManager` 밖, **애플리케이션 레벨에서 공유되는 엔티티 캐시**  
  - Hibernate의 2차 캐시 기능, EhCache/Redis/인메모리 등을 얹어서 사용

흐름을 텍스트로 정리하면 대략 이렇게 된다.

1. 애플리케이션 코드에서 `find()`/`getReference()`를 호출한다.
2. JPA는 **먼저 1차 캐시(영속성 컨텍스트)**에서 엔티티를 찾는다.
3. 1차 캐시에 없으면, **2차 캐시**를 조회한다. (2차 캐시가 활성화 되어 있다면)
4. 2차 캐시에도 없으면, **DB에서 조회**해서 엔티티를 만든다.
5. 조회된 엔티티는 1차 캐시에 등록되고, 필요하다면 2차 캐시에도 저장된다.
6. 트랜잭션이 끝나면 1차 캐시는 날아가지만, 2차 캐시는 남는다.

한 줄로 줄이면 이렇게 볼 수 있다.

> **요청/트랜잭션 스코프의 1차 캐시 → 애플리케이션 스코프의 2차 캐시 → DB**

이제 각 캐시를 조금 더 사람처럼 소개해 보자.

---

## 3. 1차 캐시 – 현재 트랜잭션의 “진실된 뷰”

JPA 1차 캐시는 영속성 컨텍스트의 일부다.  
그리고 영속성 컨텍스트는 보통 **트랜잭션 단위**로 엮인다.

즉,

- 트랜잭션이 시작될 때 영속성 컨텍스트(1차 캐시)가 만들어지고  
- 트랜잭션이 끝나면 함께 정리된다.

### 3-1. 1차 캐시가 해주는 일

1. **같은 엔티티를 두 번 조회해도, 쿼리는 한 번만 나가게 한다**

```java
User u1 = em.find(User.class, 1L); // DB 조회 + 1차 캐시에 저장
User u2 = em.find(User.class, 1L); // 1차 캐시에서 즉시 반환, 쿼리 없음

assert u1 == u2; // 같은 인스턴스
```

2. **변경 감지(Dirty Checking)의 기준이 되는 스냅샷을 관리한다**
   - 1차 캐시는 “엔티티의 현재 상태”를 들고 있고  
   - 내부적으로 “초기 상태 스냅샷”과 비교해서 변경 여부를 판단한다.

3. **트랜잭션 내에서의 정합성을 보장한다**
   - 같은 트랜잭션 안에서는 “같은 ID의 엔티티는 항상 같은 인스턴스”라는 가정을 유지해 준다.  
   - DB에서 값이 동시에 바뀌더라도, 현재 트랜잭션이 보고 있는 뷰는 “자기만의 일관성”을 유지한다.

### 3-2. 1차 캐시의 특징을 한 줄로 요약하면

- 스코프: **트랜잭션(영속성 컨텍스트) 범위**  
- 책임: 변경 감지, 동일성 보장, 트랜잭션 내 정합성  
- 수명: 트랜잭션 종료 시 함께 사라짐  
- 설정: **항상 활성화** (JPA의 기본 동작, 끌 수 없다)

그래서 1차 캐시는 “옵션”이 아니라 **JPA를 쓰는 순간 기본으로 따라오는 설계 요소**에 가깝다.

---

## 4. 2차 캐시 – 여러 트랜잭션이 공유하는 “엔티티 스냅샷”

2차 캐시는 이야기 구조가 조금 다르다.  
여기는 애초에 **성능을 위한 옵션**이다.

### 4-1. 2차 캐시가 하는 일

2차 캐시는 간단히 말하면,

> “여러 `EntityManager`/세션에서 **같은 엔티티를 반복 조회할 때, DB까지 가지 말고 미리 저장해 둔 스냅샷을 재사용하자**”

라는 아이디어에서 시작한다.

흐름은 다음과 같다.

1. 어떤 트랜잭션에서 `em.find(User.class, 1L)`을 호출한다.
2. 1차 캐시에 없고, 2차 캐시에도 없으면 DB에서 조회한다.
3. 조회된 결과를 1차 캐시에 저장하고, 동시에 **2차 캐시에도 저장**한다.
4. 다른 트랜잭션/세션에서 같은 엔티티를 조회할 때  
   - 1차 캐시에 없으면 2차 캐시에서 먼저 찾고,  
   - 있다면 DB까지 가지 않고 캐시된 값을 사용한다.

이때 중요한 포인트는,

- 2차 캐시는 기본적으로 **엔티티 단위의 스냅샷**을 저장한다는 것  
- 구현체(Hibernate)에 따라 **쿼리 캐시**(Query Cache)와 조합해서 쓰기도 한다는 것

### 4-2. 2차 캐시의 특징 요약

- 스코프: **애플리케이션 전체 (또는 캐시 영역 단위)**  
- 책임: DB 조회 결과 재사용, 읽기 성능 향상  
- 수명: 프로세스/캐시 인스턴스가 살아있는 동안 유지  
- 설정: **명시적으로 켜야 한다** (라이브러리/캐시 제공자 설정 포함)

그래서 2차 캐시는 “있으면 좋은 성능 옵션”이지만,  
동시에 **정합성과 복잡성을 함께 끌고 들어오는 선택**이기도 하다.

---

## 5. 1차 캐시 vs 2차 캐시 vs Redis 캐시 – 어떻게 다를까?

처음에는 “2차 캐시 = Redis 캐시”라고 단순하게 이해하려다가, 실제로 써보면서 생각이 조금 바뀌었다.

정리해 보면 세 가지는 이런 식으로 다르다.

### 5-1. 관점 1 – 누가 관리하나?

- **1차 캐시**
  - JPA 구현체(Hibernate)가 `EntityManager`/세션 내부에서 관리  
  - 애플리케이션 개발자는 직접 개입하지 않는다.

- **2차 캐시**
  - JPA 구현체 + 캐시 제공자(EhCache, Infinispan, Redis 등)가 함께 관리  
  - 어떤 엔티티를 캐시할지, TTL/전략을 어떻게 가져갈지 설정이 필요하다.

- **Redis 애플리케이션 캐시**
  - 애플리케이션 코드에서 직접 Redis를 호출해 관리  
  - 캐시 전략(Cache-Aside, Write-Through 등)을 직접 설계해야 한다.

### 5-2. 관점 2 – 무엇을 캐싱하나?

- **1차 캐시**
  - 현재 트랜잭션에서 관리 중인 **엔티티 인스턴스 자체**  
  - 변경 감지가 되는 “살아 있는 객체”

- **2차 캐시**
  - DB에서 읽어온 엔티티의 **스냅샷**  
  - 다시 1차 캐시에 로딩될 때 엔티티로 복원된다.

- **Redis 캐시**
  - 엔티티를 기반으로 하지만, 보통 **DTO/JSON/단순 구조**로 직렬화해서 저장  
  - 조회 API 응답이나 특정 비즈니스 뷰에 맞춘 구조일 때가 많다.

### 5-3. 관점 3 – 일관성/정합성의 기준은?

- **1차 캐시**
  - 트랜잭션 내에서 **가장 강한 정합성**을 제공  
  - “같은 트랜잭션 안에서 같은 ID의 엔티티는 항상 같은 상태”라는 보장이 있다.

- **2차 캐시**
  - 여러 트랜잭션이 공유하므로, **캐시 무효화 전략**에 따라 정합성 수준이 달라진다.  
  - 애플리케이션 인스턴스가 여러 대면, 클러스터 간 동기화 이슈까지 고려해야 한다.

- **Redis 캐시**
  - 완전히 애플리케이션 설계/전략에 의존한다.  
  - TTL, 쓰기 후 무효화/갱신, 스탬피드 방지 등 모든 걸 직접 결정해야 한다.

내가 내린 결론은 이렇다.

> - 1차 캐시는 **엔티티 라이프사이클/트랜잭션 정합성**을 위해  
> - 2차 캐시는 **엔티티 조회 성능 최적화**를 위해  
> - Redis 캐시는 **비즈니스 뷰/응답 성능 최적화**를 위해
>
> 서로 다른 층에서 사용하는 도구들이다.

## 6. 실제 프로젝트에서는 어떻게 선택했는지

실제 프로젝트에서 나는 대략 이런 기준으로 가져갔다.

1. **1차 캐시는 JPA를 쓰는 순간 기본값으로 신뢰**
   - “같은 트랜잭션 안에서 같은 엔티티는 한 번만 조회된다”는 가정을 적극 활용했다.  
   - 가능한 한 트랜잭션 범위를 명확하게 유지하려고 했다.

2. **2차 캐시는 제한적으로만 사용**
   - 읽기 빈도가 매우 높고, 변경이 상대적으로 적은 엔티티(예: 코드 테이블, 설정값 등)에만 적용  
   - 그 외 대부분의 도메인 엔티티에는 2차 캐시를 적용하지 않았다.  
   - 특히 다대일/일대다 연관관계까지 깊게 엮인 엔티티는 2차 캐시로 관리하기가 부담됐다.

3. **API 응답/뷰 레벨의 캐싱은 Redis로 분리**
   - “엔티티 레벨 최적화”와 “API 응답 레벨 최적화”를 분리하고 싶었다.  
   - JPA 캐시는 JPA 안에서, Redis 캐시는 서비스/뷰 레벨에서 역할을 나눴다.

4. **성능 튜닝의 기본 단위는 여전히 쿼리**
   - 캐시를 붙이기 전에, 먼저 쿼리/인덱스/페이징/조회 구조를 정리했다.  
   - 그 위에 2차 캐시, Redis 캐시를 얹는 식으로 단계적으로 접근하는 게 훨씬 안정적이었다.

---

## 7. 마무리 – “지금 내가 보고 있는 엔티티는 어디에서 왔는가?”

1차 캐시, 2차 캐시, Redis 캐시를 모두 쓰다 보면  
가장 중요한 질문은 결국 이 한 줄로 정리되는 것 같다.

> **“지금 내가 보고 있는 이 엔티티/데이터는 어디에서 온 것인가?”**

이 질문에 답할 수 있어야,

- 쿼리가 왜 이 타이밍에 나가는지  
- 변경이 어디에서 감지되는지  
- 언제 정합성 이슈가 생길 수 있는지

를 예측할 수 있다.

이 글에서 정리한 내용을 기준으로,

- 서비스 코드 한 군데를 고르고  
- 같은 엔티티를 여러 번 조회하는 부분이 있다면  
  - 1차 캐시가 어떻게 동작하고 있는지  
  - 2차 캐시/Redis와는 어떤 관계인지

를 한 번 손으로 따라가 보길 추천한다.

그 과정을 몇 번 반복하다 보면,  
JPA 캐시가 “알듯 말듯한 마법”이 아니라 **이해 가능한 설계 도구**에 더 가깝게 느껴질 것이다.
