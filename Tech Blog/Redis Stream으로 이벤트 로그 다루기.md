# Redis Stream으로 이벤트 로그 다루기

> “Pub/Sub으로는 아쉬웠던 내구성과 재처리를, Redis 안에서 어느 정도까지 가져갈 수 있을까?”

## 0. 이 글을 쓰게 된 계기

앞에서 Redis Pub/Sub을 정리하면서, 이런 느낌을 계속 받았다.

> “실시간 알림/신호용으로는 괜찮은데,  
>  이벤트를 쌓아 두고 다시 읽거나, 소비자 그룹별로 처리하고 싶을 땐 뭔가 부족하다.”

실제로도 그랬다.

- 특정 이벤트를 **나중에 다시 재처리**하고 싶을 때  
- 여러 소비자 그룹이 **각자 속도로 메시지를 소비**해야 할 때  
- “한 번 처리된 메시지와 아직 처리 중인 메시지를 구분”하고 싶을 때

Pub/Sub만으로는 설계가 어색해졌다.

그때 다시 보게 된 기능이 Redis의 **Stream**이다.  
Kafka처럼 “로그를 쌓고, 소비자 그룹 단위로 읽고, 필요하면 다시 읽을 수 있는” 구조를 Redis 안에 어느 정도까지 구현해 둔 기능이다.

이 글은,

- Redis Stream이 어떤 구조/개념을 가지고 있는지  
- Pub/Sub과 비교했을 때 어떤 점이 다르고, 어떤 점은 여전히 부족한지  
- 실제로 어떻게 쓰면 좋을지 (XADD, XREAD, Consumer Group 등)

을 정리해 보려는 시도다.

---

## 1. 왜 Redis Stream을 알아야 할까?

이미 Kafka나 MQ를 쓰고 있다면,  
굳이 Redis Stream까지 알아야 하나 싶은 생각이 들 수 있다.

실무에서 Redis Stream이 눈에 들어왔던 상황은 대략 이랬다.

1. **이미 Redis 인프라는 있는데, Kafka를 추가로 도입하긴 부담스러운 경우**
   - 내부 서비스 몇 개 사이에서만 이벤트를 주고받으면 되는 상황  
   - “메시지 유실은 싫지만, Kafka 클러스터까지는 좀 과하다”는 느낌일 때

2. **Pub/Sub으로 시작했다가 내구성과 재처리의 필요성이 올라간 경우**
   - “특정 기간 동안의 이벤트를 다시 읽고 싶다”  
   - “처리 실패한 이벤트만 모아서 재시도하고 싶다”

3. **단일 Redis 안에서 ‘가벼운 Kafka 느낌’을 내고 싶을 때**
   - 로그처럼 append-only로 쌓이고  
   - consumer group별로 offset(비슷한 개념)을 관리하고  
   - 처리 완료/미완료를 구분하고 싶을 때

물론 Redis Stream이 Kafka를 완전히 대체할 수 있는 수준은 아니다.  
그래도 “Pub/Sub ↔ 본격 메시지 브로커” 사이의 간극을 메워주는 도구로 볼 만했다.

---

## 2. Redis Stream 구조를 한 번에 훑어보기

핵심 개념만 잡아보면 대략 이렇다.

- **Stream**
  - append-only 로그.  
  - 각 항목은 `ID`(타임스탬프+시퀀스)와 필드-값 쌍으로 구성된다.
- **Entry**
  - Stream 안에 들어가는 하나의 레코드.  
  - 예: `1678881234567-0` ID에 `userId=1`, `orderId=123` 같은 필드들.
- **Consumer Group**
  - 같은 Stream을 읽는 **논리적 소비자 그룹**.  
  - 그룹마다 자신만의 소비 위치를 관리한다.
- **Consumer**
  - 그룹 안에 속한 실제 소비자.  
  - 같은 그룹 안의 여러 Consumer가 메시지를 분담해서 처리한다.
- **Pending Entries List (PEL)**
  - 각 그룹/컨슈머별로 “읽었지만 아직 처리 완료(ACK)되지 않은 메시지” 목록.

흐름을 텍스트로 그려보면:

1. Producer가 `XADD`로 Stream에 메시지를 추가한다.
2. Consumer Group이 `XREADGROUP`으로 Stream에서 메시지를 읽는다.
3. 각 Consumer는 자신이 받은 메시지를 처리한다.
4. 처리가 끝나면 `XACK`으로 해당 메시지를 “완료” 상태로 표시한다.
5. 처리 중 실패/장기 지연된 메시지는 PEL에서 다시 조회해 재할당/재처리할 수 있다.

한 줄로 줄이면,

> **“Redis Stream은 ‘저장되는 Pub/Sub + 간단한 consumer group + 재처리 지원’ 정도의 기능을 제공하는 이벤트 로그”**라고 이해했다.

## 3. 기본 명령어로 감각 잡기 (XADD, XREAD, XGROUP, XREADGROUP, XACK)

API를 다 외울 필요는 없고, 흐름을 기준으로 자주 쓰는 것만 정리해 보자.

### 3-1. 메시지 추가 – XADD

```bash
XADD mystream * userId 1 orderId 123
```

- `mystream`: Stream 이름  
- `*`: Redis가 자동으로 ID를 생성  
- 뒤에는 필드-값 쌍

결과는 예를 들어 `1678881234567-0` 같은 ID로 반환된다.

### 3-2. 단순 읽기 – XREAD

Stream을 그냥 “로그”처럼 읽고 싶다면 `XREAD`를 쓴다.

```bash
XREAD COUNT 10 STREAMS mystream 0-0
```

- `0-0`부터 시작해 최대 10개를 읽는다.
- “마지막으로 읽은 ID”를 기억해 두고, 다음 요청에서 그 ID 이후를 읽는 식으로 사용할 수 있다.

하지만 보통은 **Consumer Group**을 만들어 `XREADGROUP`을 더 많이 쓰게 된다.

### 3-3. Consumer Group 생성 – XGROUP CREATE

```bash
XGROUP CREATE mystream my-group 0-0
```

- `mystream`: Stream 이름  
- `my-group`: 그룹 이름  
- `0-0`: 그룹의 시작 오프셋

이제부터 `my-group`이라는 이름으로 Consumer들이 이 Stream을 읽을 수 있다.

### 3-4. 그룹으로 읽기 – XREADGROUP

```bash
XREADGROUP GROUP my-group consumer-1 COUNT 10 STREAMS mystream >
```

- `GROUP my-group consumer-1`: 그룹 이름과 Consumer 이름  
- `>`: “아직 이 그룹에서 읽지 않은 새 메시지”

이렇게 읽은 메시지는 해당 Consumer의 **Pending 목록**에 들어간다.

### 3-5. 처리 완료 표시 – XACK

메시지 처리가 끝나면, `XACK`으로 완료를 표시한다.

```bash
XACK mystream my-group 1678881234567-0
```

이렇게 하면 PEL에서 해당 메시지가 제거되고,  
“이 그룹 입장에서는 이 메시지는 더 이상 처리 대상이 아니다”라는 의미가 된다.

## 4. Redis Pub/Sub과 비교해서 뭐가 다를까?

Stream을 이해할 때 가장 도움이 됐던 건 **Pub/Sub과의 비교**였다.

### 4-1. 저장 vs 미저장

- **Pub/Sub**
  - 메시지를 **저장하지 않는다**.  
  - 구독자가 연결되어 있을 때만 메시지를 받는다.

- **Stream**
  - 메시지를 **Stream에 저장**한다.  
  - 나중에 다시 읽을 수 있고, 여러 그룹이 각자 속도로 소비할 수 있다.

### 4-2. 소비자 그룹/오프셋

- **Pub/Sub**
  - 소비자 그룹 개념이 없다.  
  - 채널을 구독한 모든 Subscriber가 메시지를 동시에 받는다.

- **Stream**
  - Consumer Group 단위로 “어디까지 읽었는지”를 관리한다.  
  - 같은 그룹 안에서는 Consumer들이 메시지를 나눠 가져갈 수 있다.

### 4-3. 재처리/실패 핸들링

- **Pub/Sub**
  - 재처리를 위한 내장 메커니즘이 없다.  
  - 실패/재시도를 애플리케이션에서 별도로 설계해야 한다.

- **Stream**
  - PEL을 통해 “읽었지만 ACK되지 않은 메시지”를 추적할 수 있다.  
  - 일정 시간 이상 처리되지 않은 메시지를 다시 가져와 다른 Consumer에게 재할당하는 패턴이 가능하다.

그래서 느낌상,

> - **Pub/Sub**은 “실시간 신호/알림”  
> - **Stream**은 “간단한 이벤트 로그 + 소비자 그룹 + 재처리”
>
> 정도로 역할이 나뉜다고 보면 편했다.

## 5. Redis Stream의 장단점과 사용할 때 주의점

### 5-1. 장점

1. **내구성/재처리가 Pub/Sub보다 훨씬 낫다**
   - 메시지를 Stream에 쌓아두고, 필요할 때 다시 읽을 수 있다.  
   - PEL + `XACK`/`XCLAIM`으로 실패/지연된 메시지를 재처리할 수 있다.

2. **Consumer Group으로 균등 분배가 가능하다**
   - 같은 그룹 안에서 여러 Consumer가 메시지를 분산 처리할 수 있다.  
   - 그룹마다 “자기만의 오프셋”을 관리할 수 있다.

3. **Kafka 도입 전 단계로 쓰기 좋다**
   - Redis 인프라만으로 “간단한 이벤트 로그 + 소비자 그룹”을 구축할 수 있다.  
   - 운영/모니터링 복잡도가 Kafka보다는 상대적으로 낮다.

### 5-2. 단점/주의점

1. **완전한 Kafka 대체는 아니다**
   - 파티셔닝, 스케일 아웃, 디스크 기반 장기 보관, 다양한 클라이언트/에코시스템 등  
   - 본격적인 로그/스트림 처리 파이프라인에는 Kafka가 여전히 더 적합하다.

2. **메모리/디스크 관리 전략이 필요하다**
   - Stream은 기본적으로 **쌓여만 가는 로그**이기 때문에,  
   - `MAXLEN` 옵션이나 트림(TRIM) 전략을 정하지 않으면 메모리/디스크 사용량이 계속 늘어난다.

3. **소비자 그룹 관리가 생각보다 신경 쓸 부분이 많다**
   - 장기간 살아 있지 않은 Consumer의 PEL,  
   - “유실된 Consumer”의 메시지를 다른 Consumer에게 넘기는 로직(`XCLAIM`) 등  
   - 운영 도구가 없는 상태에서는 직접 관리 코드를 짜야 하는 부분이 있다.

4. **at-least-once가 기본 – 중복 처리에 대비해야 한다**
   - 실패/재시도 시 **같은 메시지가 여러 번 전달될 수 있다.**  
   - Consumer 로직은 idempotent(중복 처리 안전)하게 설계하는 것이 사실상 필수다.

5. **Redis 자체의 한계를 그대로 가진다**
   - 클러스터/샤딩 구조, 네트워크 파티션, 장애 복구 시의 일관성 등  
   - “Redis를 어떻게 운영하느냐”에 따라 Stream의 신뢰성도 영향을 받는다.

## 6. 실제로는 어디에 쓰면 좋을까?

내가 보기에 Redis Stream이 잘 맞는 지점은 이런 곳들이다.

1. **내부 서비스 간 이벤트 로그**
   - 주문/포인트/알림/통계 서비스끼리의 이벤트 전달  
   - 아직 외부로 나가지 않는, 비교적 단순한 도메인 이벤트들

2. **간단한 작업 큐 / 잡 큐**
   - 특정 작업을 비동기로 처리하고 싶을 때  
   - 실패/재시도, 지연된 메시지 재처리가 필요한 경우

3. **Pub/Sub에서 한 단계 올라가고 싶을 때**
   - 기존 Pub/Sub 기반 알림 시스템에 “재처리/로그 보존”을 추가하고 싶을 때  
   - 완전히 Kafka로 넘어가기 전, Redis 인프라에서 실험해 보는 용도

반대로, 아래 같은 요구사항이라면 처음부터 Kafka·MQ 쪽을 고민하는 게 낫다고 생각했다.

- 장기간(며칠~몇 달) 이벤트를 보관해야 하는 경우  
- 수십~수백 개 팀/서비스가 같은 이벤트 스트림을 소비하는 대규모 환경  
- 스트림 처리/윈도우 집계/복잡한 파티셔닝 전략이 중요한 시스템

## 7. 마무리 – “Pub/Sub ↔ Kafka” 사이의 괜찮은 중간 지점

Redis Stream을 써보면서 느낀 포인트는 이거였다.

> “Redis Pub/Sub이 너무 가볍고, Kafka가 너무 무겁게 느껴질 때  
>  그 사이 어딘가에 놓을 수 있는 현실적인 중간 지점이 Redis Stream이다.”

이 글을 정리한 기준으로,

- 지금 시스템에서 “실시간 Pub/Sub으로는 아쉽고, 그렇다고 Kafka를 들이기엔 과한 곳”이 있는지  
- 그 지점에 Redis Stream을 한 번 얹어 보면서  
  - 메시지 로그를 어떻게 관리할지  
  - consumer group/재처리/중복 처리 전략을 어떻게 가져갈지

를 실험해 보면 좋을 것 같다.

그 과정을 거치면,

- “우리가 메시지 브로커에서 정말로 필요한 기능이 무엇인지”  
- “어디까지는 Redis로 충분하고, 어디부터는 Kafka가 필요할지”

를 훨씬 명확하게 말할 수 있게 되고,  
도구 선택이 기술이 아니라 **요구사항 기반의 설계 선택**에 더 가까워진다는 느낌을 받게 된다.
