# CORS (Cross-Origin Resource Sharing)

> 태그: `#network` `#http` `#cors` `#browser-security`<br>
> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

CORS는 브라우저의 동일 출처 정책(SOP) 때문에 막히는 교차 출처 요청을, 서버가 응답 헤더로 "허용 범위"를 명시해 풀어주는 규칙으로, 서버가 아니라 브라우저가 지키는 보안 모델이라 서버-서버 통신에는 보통 적용되지 않는다.

## 특징 / 상세

### Origin과 SOP

브라우저는 출처가 다른 리소스 접근을 기본적으로 제한한다.

- Origin: `scheme + host + port`
- 제한 대상: 대표적으로 XHR/fetch 기반 요청

### Simple request vs Preflight

조건이 단순하면 바로 보내고, 복잡하면 먼저 허용 여부를 묻는다.

- preflight: 브라우저가 `OPTIONS`로 "이 요청 해도 되나?"를 확인
- 서버가 허용하면 실제 요청을 보낸다.

### credentials(쿠키/인증정보) 주의

쿠키를 포함한 교차 출처 요청은 허용 조건이 더 엄격하다.

- 서버는 `Access-Control-Allow-Credentials` 등을 적절히 설정해야 한다.
- 임의의 출처(`*`)와 credentials 조합은 위험하거나 금지되는 경우가 많다.

### 자주 헷갈리는 포인트

CORS는 서버가 아니라 브라우저가 지키는 정책이다.

- 서버는 요청을 처리할 수 있어도, 브라우저가 응답을 막을 수 있다.
- 그래서 "서버는 200을 줬는데 프론트는 실패처럼 보이는" 상황이 생긴다.

### 실전 포인트

문제는 보통 CORS 설정의 한 줄이 아니라, 출처와 credentials 조합이다.

- 개발 환경과 운영 환경의 origin이 다르면 자주 걸린다.
- preflight가 생기는 조건을 이해하면 디버깅이 쉬워진다.
