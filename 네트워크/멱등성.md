# Idempotency (멱등성)

> 태그: `#network` `#http` `#idempotency`<br>
> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

멱등성은 같은 요청을 여러 번 보내도 결과(서버 상태)가 한 번 보낸 것과 같은 성질로, HTTP 메서드마다 멱등 여부가 정의되어 있지만(`GET`, `PUT`, `DELETE`는 멱등, `POST`는 비멱등) 재시도가 잦은 분산 환경에서는 멱등성 키 같은 별도 보완이 필요하다.

## 특징 / 상세

### 왜 중요한가

네트워크는 응답을 못 받는 상황(타임아웃, 끊김)이 흔하고, 이때 클라이언트는 재시도를 한다.

- 멱등한 요청이면 재시도해도 안전하다.
- 멱등하지 않은 요청을 재시도하면 중복 처리(중복 결제, 중복 생성 등)가 생길 수 있다.

### HTTP 메서드와 멱등성

| 메서드 | 멱등 여부 | 이유 |
|---|---|---|
| GET | 멱등 | 조회만 하므로 여러 번 호출해도 상태 변화 없음 |
| PUT | 멱등 | 같은 값으로 덮어쓰므로 여러 번 호출해도 결과 동일 |
| DELETE | 멱등 | 이미 삭제된 것을 다시 삭제해도 "삭제된 상태"는 동일 |
| POST | 비멱등 | 보통 새 리소스를 생성하므로 호출마다 새로운 결과가 생김 |

### 멱등성만으로 부족한 경우

POST처럼 원래 비멱등인 동작도 "중복 생성 방지"가 필요할 때가 있다(결제, 주문 등).

- 이때 `Idempotency Key`를 클라이언트가 생성해 요청에 포함시킨다.
- 서버는 같은 키로 들어온 요청을 "이미 처리한 요청"으로 인식해 중복 처리를 막는다.
- 또는 "중복 생성"을 막는 유니크 제약(키/인덱스)을 DB에 걸어두는 방식도 쓴다.

### 자주 헷갈리는 포인트

멱등성은 "응답이 같다"가 아니라 "최종 상태가 같다"는 뜻에 가깝다.

- 같은 `DELETE`를 여러 번 보내도 최종 상태는 "없음"으로 같다. 다만 첫 호출은 `200`/`204`, 이후 호출은 이미 없으므로 `404`를 반환할 수 있어 응답 자체는 다를 수 있다 — 그래도 멱등하다고 본다.
- 같은 `POST`는 보통 새 리소스를 더 만들 수 있어서 멱등이 아니다.

### 실전 포인트

재시도 로직을 설계할 때는 멱등성부터 확인한다.

- 멱등한 메서드는 그대로 재시도해도 된다.
- 비멱등한 메서드는 멱등성 키나 별도 중복 방지 로직이 필요하다.
