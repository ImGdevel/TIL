# CSRF (Cross-Site Request Forgery)

> 태그: `#network` `#security` `#csrf` `#cookie`<br>
> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

CSRF는 사용자가 로그인된 상태(쿠키 기반 인증)를 악용해, 공격자가 의도한 요청이 "사용자 권한으로" 전송되게 만드는 공격으로, 브라우저가 쿠키를 자동으로 첨부한다는 점을 이용하므로 서버가 "이 요청이 진짜 사용자 의도인가"를 확인해야 막을 수 있다.

## 특징 / 상세

### 왜 발생하나

공격자는 사용자를 속여 요청을 보내게 만들고, 브라우저는 쿠키를 자동으로 붙인다.

- 사용자는 이미 서비스에 로그인되어 있다.
- 공격자는 사용자의 브라우저로 "상태 변경 요청"을 날리게 유도한다.
- 서버가 방어하지 않으면 요청은 정상 사용자 요청처럼 처리될 수 있다.

### 방어 방법

"요청 출처/의도"를 검증하거나, 쿠키 자동 첨부를 줄인다.

- `CSRF token` 사용(서버가 발급, 요청마다 검증)
- `SameSite` 쿠키 설정으로 교차 사이트 요청에서 쿠키 첨부 제한
- `Origin`/`Referer` 검증(보조 수단)

### 자주 나오는 포인트

- `JWT`를 Authorization 헤더로만 보내는 구조라면, 쿠키 자동 첨부가 아니므로 CSRF 위험이 줄어들 수 있다.
- 하지만 XSS에는 여전히 취약할 수 있어, 둘을 분리해서 본다.

### 자주 헷갈리는 포인트

`CORS`와 `CSRF`는 막는 대상이 다르다.

- CORS: 다른 출처 응답을 브라우저가 읽게 할지
- CSRF: 로그인된 상태를 악용한 요청을 막을지

### 실전 포인트

쿠키 기반 인증은 CSRF 방어를 기본 세트로 봐야 한다.

- `SameSite`만으로 충분한지 여부는 서비스 특성에 따라 다르다.
- 상태 변경 요청에는 토큰 검증이나 출처 검증을 더하는 편이 안전하다.
