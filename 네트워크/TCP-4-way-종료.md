# TCP 4-way Close

> 태그: `#network` `#tcp` `#connection-close`<br>
> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

TCP 연결 종료는 양방향 통신을 각각 끊어야 해서 FIN→ACK→FIN→ACK 4단계가 필요하며, 마지막 ACK를 보낸 쪽은 상대가 ACK를 못 받았을 경우의 재전송에 대응하기 위해 TIME_WAIT 상태로 잠시 대기한다.

## 특징 / 상세

### 종료 흐름

TCP는 "연결"이 아니라 양방향 바이트 스트림이라, 어느 쪽이든 먼저 종료를 시작할 수 있다. 클라이언트가 먼저 시작하는 경우가 흔하지만 일반화하면 A/B 양쪽 누구든 시작할 수 있다.

```
A                              B
   |---------FIN----------->|   "더 보낼 데이터 없음"
   |<--------ACK------------|   확인
   |<--------FIN------------|   B도 종료 요청
   |---------ACK----------->|   확인 → TIME_WAIT 진입
```

| 단계 | 보내는 쪽 | 내용 |
|---|---|---|
| 1 | A | `FIN` |
| 2 | B | `ACK` |
| 3 | B | `FIN` |
| 4 | A | `ACK` |

### 왜 4번인가

TCP는 양방향 연결이라 양쪽 모두 "더 보낼 데이터 없음"을 선언해야 한다.

- A가 `FIN`을 보냈다고 해서 B도 즉시 끊는 것이 아니다.
- B는 자신이 보낼 데이터를 마무리한 뒤 별도로 `FIN`을 보낸다.
- 그래서 3-way handshake와 달리 종료는 한 단계가 더 필요하다(`SYN+ACK`처럼 합쳐지지 않는 경우가 많음).

> half-close라고 해서 한쪽은 종료, 다른 쪽은 계속 데이터를 보내는 상태도 가능하다(B가 FIN을 늦게 보내는 이유).

### TIME_WAIT과의 연결

마지막 `ACK`를 보낸 쪽(보통 먼저 종료를 시작한 쪽)은 곧바로 소켓을 닫지 않고 `TIME_WAIT` 상태로 일정 시간 대기한다.

- 이유: 마지막 `ACK`가 손실되어 상대가 `FIN`을 재전송할 경우에 대응하기 위함
- 자세한 내용은 [TIME-WAIT](TIME-WAIT.md) 참고.

### 실전 포인트

연결 종료 흐름을 알고 있으면 다음을 더 잘 이해할 수 있다.

- 서버를 재시작할 때 기존 연결들이 즉시 사라지지 않는 이유
- `TIME_WAIT` 소켓이 쌓이는 이유와 그게 왜 문제가 될 수 있는지
