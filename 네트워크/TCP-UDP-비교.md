# TCP vs UDP

> 태그: `#network` `#tcp` `#udp` `#transport-layer`<br>
> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

TCP는 연결을 맺고 순서/신뢰성을 보장하는 프로토콜이고, UDP는 연결 없이 보내는 가벼운 프로토콜로, 신뢰성이 필요한지 최신성/지연이 더 중요한지에 따라 선택 기준이 갈린다.

## 특징 / 상세

### TCP

- 연결형: 3-way handshake로 연결을 맺음
- 신뢰성: 순서 보장, 손실 시 재전송, 흐름/혼잡 제어
- 비용: 연결 수립/유지 오버헤드, 헤더가 더 큼

### UDP

- 비연결형: 핸드셰이크 없이 바로 전송
- 신뢰성 보장 없음: 순서/도착 보장이 없고, 손실되면 그냥 사라짐
- 비용: 오버헤드가 적어 빠르고 가벼움

### 자주 나오는 포인트

- "UDP는 신뢰성이 없다"는 말이 "UDP는 무조건 불안정하다"는 뜻은 아니다. 애플리케이션이 직접 신뢰성을 구현할 수도 있다(예: QUIC).
- 신뢰성이 없다는 특성을 오히려 장점으로 쓰는 경우가 있다(최신 데이터가 중요하고 오래된 데이터는 버려도 되는 경우). 자세한 내용은 [UDP-최신성](UDP-최신성.md) 참고.

### 선택 기준

```
데이터 정확성/순서가 중요    → TCP
  예: 파일 전송, 웹 페이지 로딩, 결제

지연/최신성이 더 중요         → UDP
  예: 실시간 영상/음성, 게임, DNS 조회

구현 복잡도도 고려          → 신뢰성이 필요한데 TCP를 못 쓰는 상황이면
  애플리케이션이 UDP 위에 재전송/순서 보장을 직접 구현해야 함 (예: QUIC)
```

선택 기준은 정확성/지연 두 가지만이 아니라 구현 복잡도까지 세 가지로 본다. TCP를 쓰면 신뢰성 로직을 전송 계층이 대신 해주지만, UDP를 쓰고 신뢰성이 필요하면 그 복잡도를 애플리케이션이 떠안아야 한다.

### 실전 포인트

선택은 "신뢰성 vs 지연" 사이의 트레이드오프로 본다.

- 신뢰성이 필요 없는 게 아니라, "늦게 정확한 것"보다 "빨리 대략적인 것"이 더 가치 있는 도메인인지를 본다.

## 트레이드오프

| 항목 | TCP | UDP |
|---|---|---|
| 일관성 | 순서/도착 보장 (높은 일관성) | 순서/도착 보장 없음 (낮은 일관성) |
| 가용성 | 연결 수립 실패 시 통신 불가 | 연결 개념이 없어 한쪽 패킷 손실이 다른 패킷에 영향 적음 |
| 지연 | 핸드셰이크 + 재전송으로 지연 발생 가능 | 핸드셰이크 없이 즉시 전송, 지연 낮음 |
| 비용 | 헤더/상태 관리 오버헤드 큼 | 헤더가 가볍고 상태 관리 불필요 |
| 운영부담 | 연결 상태(TIME_WAIT 등) 관리 필요 | 신뢰성이 필요하면 애플리케이션이 직접 구현해야 함 |
