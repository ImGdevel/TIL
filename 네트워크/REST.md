# REST (Representational State Transfer)

> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

REST는 리소스를 URI로 표현하고 HTTP 메서드로 행위를 표현하는 아키텍처 스타일로, Stateless 제약을 포함한 여러 제약조건을 따를 때 "RESTful"하다고 부르며 메서드 의미를 지키는 것이 설계의 핵심이다.

## 특징 / 상세

### 리소스 중심 설계

URI는 "무엇"(리소스)을 나타내고, 메서드는 "무엇을 할지"(행위)를 나타낸다.

```
GET    /users        모든 사용자 조회
GET    /users/1      특정 사용자 조회
POST   /users        사용자 생성
PUT    /users/1      사용자 전체 수정
DELETE /users/1      사용자 삭제
```

- URI에는 동사를 넣지 않는다(`/getUsers` 대신 `GET /users`).
- 리소스는 명사로, 행위는 HTTP 메서드로 표현한다.
- 하위 리소스도 같은 방식으로 표현한다: `/orders/1001/items` (1001번 주문의 아이템 목록).

### HTTP 메서드의 의미

| 메서드 | 의미 | 멱등성 |
|---|---|---|
| GET | 조회 | 멱등 |
| POST | 생성 | 비멱등 |
| PUT | 전체 수정(덮어쓰기) | 멱등 |
| PATCH | 부분 수정 | 비멱등(구현에 따라 다름) |
| DELETE | 삭제 | 멱등 |

### Stateless 제약

REST는 서버가 클라이언트의 상태를 기억하지 않아야 한다는 제약을 포함한다.

- 매 요청은 필요한 정보(인증 등)를 스스로 포함해야 한다.
- 이 제약 덕분에 서버를 수평 확장하기 쉬워진다.

### 자주 헷갈리는 포인트

REST는 "표준"이 아니라 "스타일/제약 모음"이다.

- 모든 제약을 완벽히 지켜야만 REST API라고 부르는 것은 아니며, 현업에서는 "RESTful한 정도"로 이야기하는 경우가 많다.
- URI에 동사가 들어가거나 PUT/PATCH를 혼용하는 등 변형이 흔하다.

### 실전 포인트

리소스 명명과 메서드 의미를 일관되게 지키는 것이 협업에서 가장 중요한 부분이다.

- 같은 리소스에 대해 메서드 의미를 다르게 쓰면(예: GET으로 삭제) 캐싱/재시도 전략이 깨진다.
- `/createUser`보다는 `/users`(POST), `/getOrder`보다는 `/orders/{id}`(GET)처럼 URI에서 동사를 빼고 메서드로 행위를 표현하면 설계가 읽기 쉬워진다.
