# [네트워크] Cookie / Session / JWT

## 한 줄 요약

> <u>`Cookie`는 브라우저가 저장/전송하는 작은 데이터이고, `Session`은 서버에 상태를 두고 쿠키로 세션 식별자를 전달하며, `JWT`는 서명된 토큰을 클라이언트가 들고 다니는 방식이다.</u>

무엇을 어디에 저장하느냐(클라이언트 vs 서버)와 폐기/회수 전략이 핵심 차이다.

핵심 키워드: `Set-Cookie`, `session id`, `stateless`, `signature`, `refresh token`

<br /><br />

---

## 1. Cookie

> <u>브라우저가 서버 도메인 기준으로 저장하고, 조건이 맞으면 요청에 자동으로 붙인다.</u>

- 서버가 `Set-Cookie`로 내려주고, 이후 요청에 `Cookie` 헤더로 전송된다.
- 보안 옵션이 중요하다. (`HttpOnly`, `Secure`, `SameSite`)

<br /><br />

---

## 2. Session

> <u>인증 상태를 서버에 저장하고, 클라이언트는 “세션 키”만 가진다.</u>

- 클라이언트: 보통 쿠키에 `session id` 저장
- 서버: `session id -> 사용자/권한` 같은 상태를 저장
- 장점: 서버가 세션을 즉시 폐기할 수 있다.
- 단점: 서버 확장 시 세션 공유(스티키/저장소) 고민이 생긴다.

<br /><br />

---

## 3. JWT

> <u>서명된 토큰에 정보를 담아 클라이언트가 보관하고, 서버는 서명 검증으로 신뢰한다.</u>

- 장점: 서버가 상태를 덜 가지므로 확장에 유리할 수 있다.
- 단점: 토큰이 유출되면 만료 전까지 악용될 수 있다.
- 보완: 짧은 만료 + `refresh token`, 또는 서버측 블랙리스트/회수 전략
