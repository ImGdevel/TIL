# HTTP Stateless(무상태성)

> 태그: `#network` `#http` `#stateless` `#rest`<br>
> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

HTTP는 기본적으로 이전 요청의 상태를 기억하지 않는 무상태(stateless) 프로토콜로, 로그인 유지처럼 보이는 동작은 쿠키/세션/토큰 같은 별도 메커니즘이 상태를 흉내 내는 것일 뿐 프로토콜 자체가 상태를 갖는 것은 아니다.

## 특징 / 상세

### Stateless의 의미

서버는 각 요청을 독립적으로 처리하며, 이전 요청 정보를 자동으로 유지하지 않는다.

- 매 요청은 그 자체로 완전한 정보를 담아야 한다.
- 서버가 클라이언트별 상태를 메모리에 들고 있지 않아도 동작 가능하다.

### 그런데 로그인은 어떻게 유지되나

상태가 "유지되는 것처럼" 보이는 이유는 클라이언트가 식별 정보를 매 요청에 같이 보내기 때문이다.

- 쿠키에 세션 ID를 담아 매 요청마다 전송
- 또는 토큰(JWT 등)을 Authorization 헤더로 매 요청마다 전송
- 서버는 "이 요청 자체"만 보고 식별 정보로 사용자를 판단한다.

### REST와의 연결

REST는 Stateless를 제약조건 중 하나로 삼는다.

- 서버가 클라이언트 상태를 기억하지 않아야 한다는 원칙
- 이 원칙이 지켜지면 서버 확장(스케일 아웃)이 쉬워진다.

### 실전 포인트

"로그인 상태 유지"는 HTTP가 상태를 가져서가 아니라, 클라이언트가 매번 식별자를 들고 오기 때문이다.

- 이 식별자를 어디에 저장하고(쿠키/로컬스토리지), 어떻게 검증하는지가 실제 설계의 핵심이다.
