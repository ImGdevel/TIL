# Cookie / Session / JWT

> 태그: `#network` `#auth` `#cookie` `#session` `#jwt`<br>
> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

Cookie는 브라우저가 저장/전송하는 작은 데이터이고, Session은 서버에 상태를 두고 쿠키로 세션 식별자를 전달하며, JWT는 서명된 토큰을 클라이언트가 들고 다니는 방식으로, 무엇을 어디에 저장하느냐(클라이언트 vs 서버)와 폐기/회수 전략이 핵심 차이다.

## 특징 / 상세

### Cookie

브라우저가 서버 도메인 기준으로 저장하고, 조건이 맞으면 요청에 자동으로 붙인다.

- 서버가 `Set-Cookie`로 내려주고, 이후 요청에 `Cookie` 헤더로 전송된다.
- 보안 옵션이 중요하다. (`HttpOnly`, `Secure`, `SameSite`)

### Session

인증 상태를 서버에 저장하고, 클라이언트는 "세션 키"만 가진다.

- 클라이언트: 보통 쿠키에 `session id` 저장
- 서버: `session id -> 사용자/권한` 같은 상태를 저장
- 장점: 서버가 세션을 즉시 폐기할 수 있다.
- 단점: 서버 확장 시 세션 공유(스티키/저장소) 고민이 생긴다.

### JWT

서명된 토큰에 정보를 담아 클라이언트가 보관하고, 서버는 서명 검증으로 신뢰한다.

- 장점: 서버가 상태를 덜 가지므로 확장에 유리할 수 있다.
- 단점: 토큰이 유출되면 만료 전까지 악용될 수 있다.
- 보완: 짧은 만료 + `refresh token`, 또는 서버측 블랙리스트/회수 전략

### 자주 헷갈리는 포인트

Cookie는 저장/전송 방식, Session은 서버 상태 방식, JWT는 토큰 표현 방식이다.

- 셋은 같은 축이 아니다. 서로 조합될 수 있다.
- 예를 들어 세션 ID를 쿠키에 담을 수 있고, JWT도 쿠키로 보낼 수 있다.

### 실전 포인트

보안은 저장 위치보다 "전송 방식과 만료/회수 전략"이 더 중요할 때가 많다.

- 쿠키 기반이면 `HttpOnly`, `Secure`, `SameSite`를 먼저 확인한다.
- JWT 기반이면 만료, 재발급, 폐기 전략을 먼저 설계한다.

## 트레이드오프

### Session vs JWT

| 항목 | Session | JWT |
|---|---|---|
| 상태 보관 위치 | 서버 (session store) | 클라이언트 (토큰 자체) |
| 즉시 폐기 | 가능 (서버에서 삭제) | 어려움 (만료 전까지 유효, 블랙리스트 추가 구현 필요) |
| 서버 확장 | 세션 공유 필요 (Redis 등) | 서버가 상태 없어 확장 용이 |
| 유출 시 위험 | 세션 ID만 무효화하면 차단 | 만료 전까지 토큰 자체로 악용 가능 |

| 항목 | 내용 |
|---|---|
| 일관성 | Session은 서버 상태가 단일 진실 소스, JWT는 클라이언트가 들고 있는 값이 검증 시점까지 진실로 취급됨 |
| 가용성 | Session은 세션 저장소 장애 시 인증 전체 영향, JWT는 서버 상태 없어 저장소 장애 영향 적음 |
| 지연 | Session은 매 요청마다 저장소 조회 필요, JWT는 서명 검증만으로 충분(조회 불필요) |
| 비용 | Session은 세션 저장소(Redis 등) 운영 비용 추가, JWT는 별도 저장소 불필요 |
| 운영부담 | Session은 저장소 확장/공유 설계 필요, JWT는 폐기 전략(블랙리스트, 짧은 만료+refresh) 설계 필요 |
