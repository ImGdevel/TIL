# HTTP/1.1 vs HTTP/2 (Keep-Alive)

> 태그: `#network` `#http` `#http2` `#performance`<br>
> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

HTTP/1.1은 Keep-Alive로 TCP 연결을 재사용하지만 한 연결당 한 번에 하나의 요청만 처리해 Head-of-Line Blocking이 생기고, HTTP/2는 하나의 연결 위에서 여러 요청/응답을 동시에 주고받는 멀티플렉싱과 헤더 압축(HPACK)으로 이 문제를 줄인다.

## 특징 / 상세

### HTTP/1.1

- 기본은 요청/응답이 텍스트 형식
- Keep-Alive: 요청마다 TCP 연결을 새로 맺지 않고 재사용
- 파이프라이닝(pipelining): 응답을 기다리지 않고 다음 요청을 미리 보내는 것은 가능하지만, 응답은 보낸 순서대로 와야 해서 실전에서는 제약이 많아 잘 쓰이지 않는다.
- 한계: 한 연결에서 동시에 하나의 요청만 처리 가능(순차 처리)
- 결과: 여러 리소스를 빨리 받으려면 연결을 여러 개 열어야 했다(브라우저별 동시 연결 수 제한 존재).

### HTTP/2

- 바이너리 프레이밍: 메시지를 텍스트가 아닌 프레임(바이너리 단위)으로 쪼개 전송
- 멀티플렉싱: 하나의 TCP 연결 위에서 여러 요청/응답을 스트림 단위로 동시 처리
- HPACK: 헤더를 압축해 반복 전송 비용을 줄임
- 결과: 연결 수를 늘리지 않고도 동시성 문제를 완화

### 비교 포인트

| 항목 | HTTP/1.1 | HTTP/2 |
|---|---|---|
| 연결 재사용 | Keep-Alive로 재사용 | 기본적으로 연결 재사용 + 멀티플렉싱 |
| 동시 요청 처리 | 연결당 1개씩 순차 처리 | 연결당 여러 스트림 동시 처리 |
| Head-of-Line Blocking | 발생(요청 순서대로 막힘) | 애플리케이션 레벨에서는 완화(전송 레벨에서는 TCP 자체의 HOL은 남음) |
| 헤더 처리 | 매번 평문 헤더 전송 | HPACK으로 압축 |

### Keep-Alive의 의미

연결을 끊고 다시 맺는 비용(3-way handshake 등)을 줄이기 위한 장치다.

- HTTP/1.1: `Connection: keep-alive` 헤더로 명시
- HTTP/2: 연결 재사용이 기본 동작에 가까움

### 실전 포인트

HTTP/2 도입의 핵심 이유는 "연결 비용 절감 + 동시성 향상"이다.

- 다만 TCP 레벨의 패킷 손실 시 HOL 문제는 HTTP/2에서도 완전히 사라지지 않는다(이 부분은 HTTP/3·QUIC에서 다루는 영역).

## 트레이드오프

| 항목 | 내용 |
|---|---|
| 일관성 | 해당 없음 — 둘 다 단일 서버 응답을 받는 구조라 일관성 이슈는 발생하지 않음 |
| 가용성 | HTTP/2는 연결 수가 적어 연결 자체의 장애 영향 범위가 더 큼(단일 연결에 의존도가 높아짐) |
| 지연 | HTTP/2가 멀티플렉싱으로 다수 리소스 동시 요청 시 지연이 더 낮음 |
| 비용 | HTTP/2는 연결 수가 적어 서버 측 연결 관리 비용(소켓/메모리)이 낮음 |
| 운영부담 | HTTP/2는 TLS 사용이 사실상 필수라 인증서 관리 부담이 추가됨 |
