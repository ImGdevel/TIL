# 테스트 전략 발표 대본(스토리) – 우리 프로젝트의 테스트 흐름·컨벤션이 만들어진 이유

> Type: TALK  
> 목적: `docs/test` 아래에 흩어진 테스트 전략/컨벤션/평가/참고 요약을 **기술 발표용 스토리**로 하나로 엮는다.  
> 범위: 본 문서는 `docs/test/*.md` 전체 내용을 기반으로 작성되었고, 마지막에 “반영 체크리스트”로 출처를 명시한다.

## 프롤로그: “테스트를 안 믿는 팀”에서 “테스트가 길잡이인 팀”으로

처음 이 프로젝트에서 테스트를 다루기 시작했을 때, 우리는 사실 “테스트가 중요하다”는 말을 이미 알고 있었습니다.  
하지만 실제 개발 루프에서 테스트는 자주 이런 모습이었습니다.

- 실행이 느려서 **자주 돌리기 어렵고**(Fast 위반)
- 실행 순서나 환경에 따라 **가끔씩 깨지고**(Independent/Repeatable 위반)
- 실패해도 “어디가 문제인지”보다 “뭐가 또 설정이야?”가 먼저 떠오르는(자가 검증 부족)
- 시간이 지나면 “리팩터링을 지켜주는 보험”이 아니라 “리팩터링을 막는 족쇄”가 되어버리는

그래서 우리가 처음 한 일은 “테스트를 늘리자”가 아니라, **테스트를 ‘믿을 수 있는 시스템’으로 만들자**였습니다.

이 문서는 그 과정의 이야기입니다.  
왜 레이어를 나눴는지, 왜 태그/태스크를 고정했는지, 왜 @MockBean/@MockitoBean을 조심하는지, 왜 픽스처를 `java-test-fixtures`로 구조화했는지, 왜 커버리지 70%를 ‘하한선’으로 삼았는지—그 이유를 “발표 흐름”으로 정리합니다.

그리고 이 이야기의 숨은 전제는 하나입니다.

> 테스트 코드는 “코드 조각”이 아니라 “개발 루프”다.  
> 빠르게 돌려서(FAST), 반복 가능해야 하고(REPEATABLE), 실패하면 바로 이해되어야 한다(SELF-VALIDATING).

## 1장: 첫 번째 사건 – “테스트가 환경을 요구하기 시작했다”

어느 날, 새로 합류한 사람이 `./gradlew test`를 돌렸습니다.  
그런데 코드는 맞는데 테스트가 깨졌습니다. 이유는 코드가 아니라 **외부 의존성**이었습니다.

- Cloudinary/OAuth/OpenAI 같은 외부 설정이 없으면 테스트가 실패
- “로컬에만 있는 환경 변수”가 없으면 재현 불가

이 사건이 우리에게 준 교훈은 단순합니다.

1) 테스트는 외부 환경을 “요구”하면 안 된다.  
2) 테스트는 외부 시스템을 “대체”할 수 있어야 한다.  

그래서 `docs/test/TESTING_QUICKSTART.md`에서 “테스트 실행 커맨드”를 고정하고, 외부 의존성을 보호하는 원칙을 명문화했습니다.

- 기본 실행: `./gradlew test`
- 빠른 실행(권장): `./gradlew :app-api:unitTest`, `./gradlew :app-api:integrationTest`
- 원칙: 단위는 외부 없이, 통합은 경계 명확히, 컨트롤러는 슬라이스 일관성 유지
- 외부 의존성은 테스트에서 No-Op/Fake로 대체(예: `NoOpImageStorageService`)

여기서부터 우리의 테스트 전략은 “테스트 코드를 더 쓰는 것”이 아니라  
**테스트가 돌아가야 하는 조건을 명확히 하는 것**에서 출발합니다.

## 2장: 두 번째 사건 – “테스트가 많아질수록, 무엇을 언제 돌릴지 몰라졌다”

테스트가 늘어나면 생기는 아주 현실적인 문제가 있습니다.

> “지금 수정한 건 작은데… 전체 테스트를 다 돌려야 하나?”

이 질문에 답이 없으면, 테스트는 곧 개발 루프에서 밀려납니다.  
그래서 우리는 `docs/adr/ADR_TESTING_TAGS_AND_GRADLE_TASKS.md`(2026-01-01, Proposed)에서 기준을 고정했습니다.

핵심 결정은 3가지입니다.

1) 레이어별 테스트는 **메타 어노테이션**으로 고정한다.  
2) 레이어별 테스트는 **JUnit Tag**로 구분한다.  
3) Gradle에서 **선택 실행 태스크**를 제공한다.  

이게 중요한 이유는 단순히 “편해서”가 아닙니다.

- 태그/태스크가 없으면, 결국 느린 테스트까지 섞여서 테스트 실행 자체가 줄어든다.
- 메타 어노테이션이 없으면, `@SpringBootTest`/`@WebMvcTest`/`@DataJpaTest` 사용 기준이 흔들린다.
- 기준이 흔들리면 테스트가 “팀의 자산”이 아니라 “개인의 취향”이 된다.

그래서 우리는 “테스트를 실행하는 방법”까지도 컨벤션으로 만들었습니다.  
테스트의 시작은 코드가 아니라 **실행 흐름**이기 때문입니다.

## 3장: 세 번째 사건 – “커버리지는 올랐는데, 버그는 못 막았다”

우리는 커버리지를 보고 안심하고 싶었습니다.  
하지만 `docs/test/ORIGIN.md`가 말하듯, 커버리지 숫자 자체는 품질을 보장하지 않습니다.

이 프로젝트에서 말하는 “유효한 테스트 커버리지”는 이런 상태입니다.

- 핵심 도메인 규칙이 충분히 검증되어 실제 버그를 막는다
- 컨트롤러↔도메인 사이의 흐름이 통합 테스트로 검증된다
- 단순 DTO/설정/자동 생성 코드는 커버리지 계산 대상에서 제외한다
- 숫자 기준(최소 70%)은 **하한선 경고 장치**다(상한선 목표가 아니다)

즉, “70%”는 명예의 숫자가 아니라 **위험 신호를 감지하는 경보선**입니다.  
그래서 우리는 JaCoCo를 쓰되, 다음과 같은 대상을 제외해 “의미 없는 커버리지 놀이”를 막습니다.

- QueryDSL Q 클래스, Application 부트스트랩, Config, dto, exception 등

이 철학은 `docs/test/TEST_BEST_PRACTICES.md`의 메시지와도 맞닿아 있습니다.

- 커버리지는 참고 수단일 뿐
- 중요한 것은 “어디를 커버하느냐”
- 핵심 도메인/복잡 로직은 높은 커버리지가 가치가 있고, 단순 위임/DTO는 과집착하지 않는다

이제 우리는 커버리지를 “목표”가 아니라 “피드백 도구”로 봅니다.

## 4장: 우리가 세운 큰 틀 – “가치 중심 + 테스트 피라미드 + 레이어 책임”

여기서 토스의 메시지가 한 문장으로 정리됩니다. (`docs/test/TOSS_SERVER_TEST_STRATEGY_SUMMARY.md`)

> 은탄환은 없다. 대신, **가치 있는 테스트에 집중**해야 한다.

그래서 프로젝트 테스트 전략(`docs/test/PROJECT_TEST_STRATEGY.md`)은 다음을 목표로 삼습니다.

- 빠른 피드백 + 높은 신뢰도
- 테스트가 설계를 끌어올리는 피드백 루프
- Given/Fixture/의존성까지 구조적으로 관리

그리고 구조는 전형적인 테스트 피라미드를 **프로젝트 현실에 맞게** 적용합니다.

- 유닛 테스트: 가장 많게(도메인 규칙/서비스 분기/정책)
- 통합 테스트: 필요한 만큼(설정/쿼리/트랜잭션/보안/핵심 플로우)
- 인수(E2E): 적지만 강하게(핵심 사용자 시나리오)

이 “큰 틀”이 정해졌기 때문에, 그 다음부터는 각 레이어에서 “무엇을 테스트하고 무엇을 버릴지”를 선명하게 정의할 수 있었습니다.

## 5장: 레이어별 컨벤션 – “무엇을 테스트할지, 무엇을 포기할지”

### 5.1 도메인(Entity/Policy) 테스트 – 가장 빠르고 가장 단단한 안전망

도메인 테스트 컨벤션(`docs/test/DOMAIN_TEST_CONVENTION.md`)은 단순합니다.

- **도메인 객체 스스로가 지켜야 하는 규칙과 상태 전이**를 검증한다
- DB/스프링/HTTP/보안은 여기로 들어오지 않는다

특히 정책(Policy) 테스트는 “입력 조건 → 허용/거부/예외”를 문서화합니다.  
이 레이어가 촘촘할수록, 서비스/통합 테스트는 “대표 시나리오”만으로도 강해집니다.

그리고 도메인 레이어에서의 중요한 경고가 하나 있습니다.

> SUT(테스트 대상) 엔티티 자체는 가능한 Fixture 대신 실제 팩토리/생성자를 직접 호출한다.  
> (Fixture가 SUT의 버그를 그대로 따라가면 테스트가 버그를 놓칠 수 있다)

이 원칙은 `docs/test/TEST-FIXTURE_V2.md`의 레이어별 픽스처 규칙과도 정렬됩니다.

### 5.2 Repository(JPA) 테스트 – “쿼리 구현”이 아니라 “쿼리 결과”를 검증한다

리포지토리 테스트 컨벤션(`docs/test/REPOSITORY_TEST_CONVENTION.md`)의 핵심은 역할 분리입니다.

- JPA 매핑, 커스텀 쿼리, 카운터/상태 변경 쿼리, cascade/orphanRemoval 같은 “데이터 접근 계약”을 검증한다
- 서비스/도메인 정책, HTTP 계약, 보안/필터는 하지 않는다
- “SQL이 몇 번 나갔는지” 같은 구현 세부는 직접 assert하지 않고, **결과와 도메인 규칙**으로 검증한다

이렇게 하면 JPQL→QueryDSL로 구현을 바꿔도 테스트는 그대로 통과해야 합니다.  
즉, 리포지토리 테스트는 “리팩터링 내성”을 품질 기준으로 삼습니다.

추가로, 현실적인 운영 규칙도 있습니다.

- 테스트 DB(H2 등)의 제약 때문에 일부 케이스는 `@Disabled`로 남길 수 있지만, **의도(왜 필요한 테스트인지)는 문서화**해 둔다
- cascade/orphanRemoval은 대표 시나리오 1~2개를 확실히 잡고, 나머지는 설계/코드리뷰로 관리한다

### 5.3 Service 단위 테스트 – 오케스트레이션과 분기/예외를 빠르게

서비스 단위 테스트 컨벤션(`docs/test/SERVICE_TEST_CONVENTION.md`)은 서비스의 정체성을 이렇게 정의합니다.

> Controller 아래, Repository/도메인 위에서 “비즈니스 플로우”를 조립하는 레이어

그래서 서비스 테스트는 다음을 검증합니다.

- 협력자 조합(도메인/리포지토리/정책 호출 흐름)
- 분기와 예외(없는 리소스, 권한 위반, 정책 위반 등)
- “비즈니스적으로 의미 있는 부작용”(카운트 증가/감소, 외부 알림 호출 등)

그리고 동시에, 다음을 **하지 않습니다**.

- JPA 동작 자체(Repository 테스트/통합 테스트로)
- HTTP 계약(Controller WebMvc 테스트로)
- 보안/필터/AOP/설정(통합 테스트로)

또 하나의 현실 규칙이 있습니다.

- 단순 위임(thin service) 수준이라면, 해당 로직이 Repository/도메인/통합 테스트로 이미 충분히 검증되는지 보고 **서비스 단위 테스트를 생략할 수 있다**

이때 조졸두가 던지는 질문(`docs/test/JOJOLDU_STUB_SERVICE_UNIT_TEST_SUMMARY.md`)이 우리에게 도움이 됐습니다.

> 서비스도 통합 테스트로만 검증하면 느리고 원인 파악이 어렵다.  
> Stub/Fake/Mock으로 외부 의존성을 끊고 빠른 단위 테스트를 만들자.

하지만 동시에 경고도 같습니다.

> 테스트 더블이 많이 필요하다면, 설계가 너무 크다는 신호일 수 있다.

이 관점은 카카오페이 Part 2(테스트는 알람)(`docs/test/KAKAOPAY_MOCK_TEST_CODE_PART2_SUMMARY.md`)로 자연스럽게 이어집니다.

### 5.4 Controller(WebMvc) 슬라이스 테스트 – “HTTP 계약”을 문서화한다

컨트롤러 WebMvc 테스트 컨벤션(`docs/test/CONTROLLER_WEBMVC_TEST_CONVENTION.md`)의 목표는 명확합니다.

- 요청 매핑/바인딩/Validation/예외 매핑/응답 포맷(ApiResponse) 등 **HTTP 레이어의 계약**을 검증한다
- 서비스/도메인 로직 자체는 여기에서 증명하려 하지 않는다

그래서 컨트롤러 테스트는 “이 입력 → 이 상태 코드 → 이 응답 구조”를 보여주는 **살아있는 API 문서**가 됩니다.

그리고 이 레이어는 Spring Boot 3.4 변화(`docs/test/SPRINGBOOT34_CONTROLLER_TEST_GUIDE.md`)와 직접 연결됩니다.

- 과거의 습관적인 `@WebMvcTest + @MockBean`은 더 이상 ‘기본값’이 아니다
- `@MockBean`/`@MockitoBean` 남발은 느려지고(컨텍스트 캐시 파괴), 불안정해질 수 있다
- 대안은 “명시적 Stub Bean 등록(@TestConfiguration/@Import + @Primary)” 혹은 더 가벼운 Standalone MockMvc

즉, 컨트롤러 테스트는 목적에 따라 3가지 무기를 나눠 쓴다는 결론으로 갑니다.

1) Standalone MockMvc: 가장 빠른 단위 테스트(컨트롤러 로직 자체)  
2) `@WebMvcTest`: MVC 인프라(Validation/Advice/직렬화)를 포함한 슬라이스 테스트  
3) `@SpringBootTest + MockMvc`: 보안/필터/빈 구성까지 포함한 통합 테스트  

우리는 이 분리를 “취향”이 아니라, **테스트 비용과 신뢰도의 트레이드오프를 관리하는 방법**으로 봅니다.

### 5.5 Integration 테스트 – “전체 플로우 + 상태 변화”로 신뢰를 올린다

통합 테스트 컨벤션(`docs/test/INTEGRATION_TEST_CONVENTION.md`)은 “무거운 테스트를 어떻게 건강하게 쓰는가”에 대한 규칙입니다.

- HTTP 기반 플로우: `@IntegrationTest` / `@IntegrationSecurityTest`
- 비HTTP 플로우: `@ServiceIntegrationTest`, 배치/잡: `@JobIntegrationTest`

여기서 핵심은 두 가지입니다.

1) 통합 테스트는 “모든 분기”를 다루지 않는다(도메인/서비스/WebMvc로 분산)  
2) 통합 테스트는 “핵심 플로우”에서 **HTTP 응답의 핵심 + DB 상태 변화**를 함께 본다  

그래서 통합 테스트는 “이 기능이 실제 앱 환경에서 처음부터 끝까지 돈다”를 보장하는 레이어가 됩니다.

## 6장: 테스트 더블 전략 – “Mock을 잘 쓰는 것”이 아니라 “Mock을 줄일 수 있는 설계”

테스트 더블은 도구인데, 어느 순간 목적이 되어버리기 쉽습니다.  
조졸두(`docs/test/JOJOLDU_AVOID_INTERNAL_ASSERTS_SUMMARY.md`)가 경고하는 핵심이 여기 있습니다.

- 내부 구현(호출 횟수/순서/중간 필드)에 집착하면 테스트는 리팩터링을 방해한다
- Mock/Spy에 과도하게 의존하면 테스트는 “변화에 대한 저항”이 된다

그래서 우리는 테스트 더블 가이드/컨벤션을 따로 만들었습니다.

- 개념 정리: `docs/test/TEST_DOUBLES_GUIDE.md`
- Spring 실무 예시: `docs/test/TEST_DOUBLES_SPRING_EXAMPLES.md`
- 우리 프로젝트 기준 선택 규칙: `docs/test/TEST_DOUBLE_CONVENTION.md`

프로젝트 기준 한 줄 요약은 이것입니다.

> 실제 객체 → Fake/Stub → Mock(Spy는 마지막 수단)

그리고 `verify()`는 “기술적 습관”이 아니라, **비즈니스적으로 의미 있는 행위**에만 허용합니다.  
이 원칙은 Comment/Like 서비스 평가(`docs/test/COMMENT_SERVICE_TEST_EVALUATION.md`)에서도 그대로 적용됩니다.

- 카운트 증감 같은 “호출 자체가 규칙”인 경우에만 verify
- 그 외에는 결과/상태 중심으로 검증

카카오페이 Part 1(`docs/test/KAKAOPAY_MOCK_TEST_CODE_SUMMARY.md`)도 같은 방향을 현실 문제로 설명합니다.

- Mock Server는 JSON/HTTP 셋업이 너무 커지면 생산성이 급격히 떨어진다
- `@MockBean` 남발은 컨텍스트 재초기화로 빌드 시간을 폭발시킨다
- 대안: `@TestConfiguration + @Primary`로 “항상 같은 Mock Bean”을 등록해 컨텍스트 재사용
- 핵심: 테스트하기 어려운 Black Box 영역이 전이되지 않게 경계를 세운다

여기서 우리는 결론을 하나 더 얻습니다.

> “Mock을 덜 쓰는 전략”은 결국 “경계를 잘 나누는 설계”에서 나온다.

## 7장: 픽스처(Fixture) – Given 지옥을 ‘구조’로 해결하기

처음 테스트가 지옥이 되는 순간은 대개 Given에서 시작합니다.

- 파라미터가 많은 엔티티/DTO 생성
- 테스트마다 반복되는 기본값
- 멀티 모듈에서 복붙되는 헬퍼
- 외부 인프라 요청/응답 매핑 코드가 테스트 본문을 덮는 상황

카카오페이 Given Part 2(`docs/test/KAKAOPAY_GIVEN_TEST_CODE2_SUMMARY.md`)가 이 문제를 아주 직관적으로 부릅니다.

> Given 지옥

그리고 해결책은 “헬퍼 함수 몇 개”에서 끝나지 않습니다.  
테스트 픽스처 가이드(`docs/test/TEST_FIXTURES_GUIDE.md`)가 말하듯, 테스트도 소프트웨어이고 **구조적으로 관리**해야 합니다.

그 구조의 중심이 Gradle `java-test-fixtures`입니다. (토스 Gradle 요약: `docs/test/TOSS_GRADLE_TEST_DEPENDENCY_MANAGEMENT_SUMMARY.md`)

- 공용 테스트 코드(Builder/Helper/Fake/Stub)를 `src/testFixtures`로 분리
- 모듈 간 공유를 `testImplementation(testFixtures(project(":...")))`로 명시
- 테스트 전용 의존성(H2 등)도 `testFixturesRuntimeOnly` + `testRuntimeOnly(testFixtures(...))`로 전파

그리고 우리 프로젝트 내부 적용 문서가 있습니다.

- “이 프로젝트에서 적용한 방법”: `docs/test/TEST-FIXTURE.md`
- “규칙만 모은 컨벤션”: `docs/test/TEST-FIXTURE_V2.md`
- “범용 가이드”: `docs/test/TEST-FIXTURE-CONVENTION.md`

핵심 규칙은 다음과 같습니다.

- 네이밍: `대상 + Fixture`, `DEFAULT_`/`UPDATED_` 상수
- 엔티티: `create(...)`, `createWithId(...)` (+ 상태가 의미 있으면 메서드명에 드러내기)
- 요청 DTO: `createRequest()`, `updateRequest()` (+ 실패/경계값은 이름으로 명시)
- 응답 DTO: 반복될 때만 Fixture 허용(기본은 핵심 필드 직접 assert)

그리고 가장 중요한 “경계”가 하나 더 있습니다.

> 비즈니스 규칙은 Fixture에 넣지 않는다.  
> Fixture는 “유효한 기본값 + 테스트 편의”까지만 맡는다.

Fixture가 도메인이 되기 시작하면, 우리는 다시 다른 지옥으로 갑니다.

## 8장: 어노테이션/테스트 인프라 – “헷갈림”을 “암기표”로 바꾸기

테스트가 늘수록 팀은 이런 질문을 반복합니다.

- “여기엔 @SpringBootTest가 맞나?”
- “@WebMvcTest에 뭘 더 Import해야 하지?”
- “@MockBean을 쓰면 왜 갑자기 느려지지?”

그래서 `docs/test/SPRING-TEST-ANNOTATIONS-GUIDE.md`는 테스트 어노테이션을 “체계”로 정리합니다.

- JUnit5: `@Test`, `@BeforeEach`, `@Nested`, `@ParameterizedTest` 등
- Mockito: `@Mock`, `@Spy`, `@InjectMocks`, `@Captor`, `@ExtendWith(MockitoExtension.class)` 등
- Spring Test 컨텍스트: `@ExtendWith(SpringExtension.class)`, `@ActiveProfiles`, `@DirtiesContext`(최소 사용) 등
- Spring Boot 테스트: `@SpringBootTest`, 슬라이스(`@WebMvcTest`, `@DataJpaTest` 등)
- Bean 교체: `@MockBean`/`@SpyBean`, 그리고 대안인 `@TestConfiguration`/`@Import`

여기에 실무에서 자주 마주치는 “운영 팁”을 덧붙이면 이렇습니다.

- `@DirtiesContext`는 컨텍스트 캐시를 깨므로 “마지막 수단”으로만 쓴다(보통은 설계를/테스트 구성을 바꾸는 게 이득)
- DB 테스트에서는 `@Transactional`/`@Rollback`, 데이터 준비/정리는 `@Sql`/`@SqlGroup` 같은 선택지가 있다
- JSON 직렬화는 `@JsonTest`, 외부 HTTP 클라이언트는 `@RestClientTest` 같은 슬라이스도 있다(필요할 때만)
- 테스트 전용 빈 교체는 `@MockBean`보다 `@TestConfiguration`+`@Primary`가 컨텍스트 재사용에 유리한 경우가 많다

그리고 토스 글에서 아주 구체적으로 짚는 “스프링 테스트 인프라 주의점”이 하나 더 있습니다. (`docs/test/TOSS_SERVER_TEST_STRATEGY_SUMMARY.md`)

- 커스텀 `TestExecutionListener`를 붙일 때 기본 리스너가 빠지지 않도록 `mergeMode = MERGE_WITH_DEFAULTS`를 의식한다  
  (기본 리스너가 빠지면 DI/트랜잭션 같은 기본 동작이 깨질 수 있다)

이 “암기표” 위에, 우리는 프로젝트 메타 어노테이션을 얹었습니다.

- `@UnitTest` → unit
- `@ControllerWebMvcTest` → (컨트롤러 슬라이스)
- `@RepositoryJpaTest` → repository
- `@IntegrationTest` / `@IntegrationSecurityTest` → integration
- `@ServiceIntegrationTest` → service-integration
- `@JobIntegrationTest` → job-integration

즉, 팀이 기억해야 하는 건 수십 개의 어노테이션 조합이 아니라,  
**“이 레이어는 이 메타 어노테이션”**이라는 단순한 룰입니다.

## 9장: 지금 우리의 테스트는 어떤 상태인가? (평가와 다음 액션)

전략은 문서로 끝나면 의미가 없습니다.  
그래서 우리는 테스트 평가 리포트(`docs/test/TEST_SUITE_EVALUATION.md`, 마지막 업데이트 2025-11-30)로 현실을 확인합니다.

요약하면:

- 잘하고 있는 부분: 리포지토리/JPA 검증, 일부 정책 주입 패턴, 통합 테스트의 DB 상태 검증, Validator 분리 테스트
- 위험 신호: 서비스 단위 테스트의 happy path 편향, 컨트롤러/인수 테스트의 오류 흐름 부족, 시큐리티 계층 테스트 부재, 정책 테스트 방치(비활성화)

그리고 Comment/Like 테스트 평가는(`docs/test/COMMENT_SERVICE_TEST_EVALUATION.md`) “Mock을 쓰되 과명세하지 않는” 좋은 패턴을 보여주면서도,

- 조회/페이징 같은 곳에서 Mock 준비 코드가 커질 수 있는 위험
- 통합 테스트와의 역할 분리 명확화 필요

같은 개선 포인트를 남깁니다.

결론적으로 다음 액션은 “테스트를 무조건 늘리기”가 아니라,

1) 시큐리티 핵심 컴포넌트 단위 테스트 보강  
2) 서비스 예외/정책/경계 케이스 보강  
3) 컨트롤러/인수 테스트에 400/401/403/404 흐름 추가  
4) 통합 테스트는 대표 플로우 + 상태 변화 중심으로 유지  
5) 비활성화된 정책 테스트 복구  

처럼 “가치/위험” 기준으로 우선순위를 두고 진행하는 것입니다(토스 전략 정렬).

## 에필로그: 한 기능을 추가할 때, 우리가 테스트를 쓰는 실제 흐름

발표를 듣는 사람이 가장 알고 싶은 건 대개 이것입니다.

> “그래서 나는 내일부터 뭘 하면 되나요?”

우리는 기능 개발을 다음 순서로 테스트합니다.

1) 도메인 규칙이 있다면: 도메인(Entity/Policy) 유닛 테스트로 먼저 문서화  
2) 서비스에 분기/예외/오케스트레이션이 있다면: `@UnitTest`로 빠르게 케이스 추가  
3) JPA 쿼리/매핑/카운터 쿼리가 추가되면: `@RepositoryJpaTest`로 결과 중심 검증  
4) API 계약이 바뀌면: `@ControllerWebMvcTest`로 HTTP 계약(Validation/예외/응답 포맷) 명시  
5) 핵심 유즈케이스라면: `@IntegrationTest`로 대표 플로우 + DB 상태 변화 1~2개를 강하게 고정  

그리고 실행은 단순합니다.

- 빠른 루프: `./gradlew :app-api:unitTest`
- 핵심 검증: `./gradlew :app-api:integrationTest`
- 전체: `./gradlew test`

이 흐름이 유지되면, 테스트는 “쓰기 귀찮은 코드”가 아니라  
**설계를 좋게 만들고, 리팩터링을 가능하게 하고, 신규 합류자에게는 가장 정확한 문서가 되는 코드**가 됩니다.

## 부록 B. 발표용 “테스트 작성 원칙” 슬라이드(요약본)

`docs/test/TEST_BEST_PRACTICES.md`와 `docs/test/INFLAB_TEST_CODE_SUMMARY.md`의 내용을 발표용으로 압축하면 다음과 같다.

- FIRST: Fast / Independent / Repeatable / Self-validating / Timely
- 이름: `what_when_then`처럼 “상황(when) + 기대(then)”가 드러나게
- 구조: Given–When–Then(또는 Arrange–Act–Assert)을 눈에 보이게
- 중복: “DRY보다 DAMP(의미/설명)”를 우선하고, 과도한 추상화는 피한다
- 케이스: 정상 1개로 끝내지 말고 경계값/예외(400/401/403/404 포함)를 반드시 넣는다
- 반복 가능성: 시간/랜덤/환경 변수 의존은 추상화(TimeProvider/Clock 등)로 고정한다
- 유지보수: 프로덕션 코드가 바뀌면 테스트도 함께 리팩터링한다(방치 금지)
