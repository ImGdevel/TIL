# 이벤트 드리븐 아키텍처(Event-Driven Architecture) 정리

> 연결된 정리본:
> - [이벤트 드리븐 아키텍처](../../../TIL.wiki/event-driven-architecture.md)


> “요청/응답으로는 점점 버티기 어려워진다. 그래서 ‘이벤트’로 시스템을 설계하기 시작했다.”

---

## 1. 이벤트 드리븐 아키텍처란?

- **시스템의 상태 변화나 비즈니스 행위를 ‘이벤트’로 표현하고,  
  그 이벤트를 발행·구독하는 방식으로 컴포넌트들이 느슨하게 결합되는 아키텍처 스타일**
- 핵심 키워드:
  - **Event**: “무슨 일이 일어났는지”에 대한 사실(Fact)
  - **Producer / Publisher**: 이벤트를 발행하는 쪽
  - **Consumer / Subscriber**: 이벤트를 구독하고 반응하는 쪽
  - **Event Broker**: 이벤트를 전달·저장·라우팅하는 중간자(Kafka, RabbitMQ 등)

> 전통적인 요청/응답(Call) 중심이 아니라, “~이 일어났다”는 신호를 중심으로 시스템을 설계한다.

---

## 2. 기본 구성 요소와 흐름

1. **이벤트 생산(Producer)**
   - 주문 생성, 결제 완료, 회원 가입 등 비즈니스 로직에서 이벤트 객체 생성
   - 이벤트 브로커(메시지 큐, 스트리밍 플랫폼)에 publish

2. **이벤트 브로커(Event Broker)**
   - Apache Kafka, RabbitMQ, AWS SNS/SQS, NATS 등
   - 이벤트를 **버퍼링/저장/전달**하고, 구독자에게 라우팅

3. **이벤트 소비(Consumer)**
   - 특정 이벤트 타입(Topic/Queue)을 구독
   - 이벤트를 수신하면, 그에 맞는 비즈니스 로직 실행
     - 예: 알림 전송, 재고 차감, 포인트 적립, 데이터 동기화

4. **이벤트 스키마(Event Schema)**
   - 이벤트의 구조(필드, 타입, 의미)를 정의
   - Avro/JSON/Protobuf + Schema Registry 등을 사용해 버전 관리하기도 함

---

## 3. 이벤트 드리븐 아키텍처의 장점

- **느슨한 결합(Loosely Coupled)**
  - Producer는 Consumer를 직접 몰라도 됨
  - 단지 “이런 이벤트를 발행한다”는 계약만 맞추면, 새로운 Consumer를 자유롭게 추가 가능

- **확장성과 유연성**
  - 특정 이벤트를 구독하는 Consumer를 수평 확장하기 쉽고
  - 새로운 기능(서비스)을 이벤트 구독자로 붙이는 방식으로 증설 가능

- **비동기 처리에 강함**
  - 응답에 직접 필요하지 않은 작업(알림 발송, 로그 적재, 통계 시스템 업데이트 등)을  
    별도 Consumer가 비동기로 처리하여 **사용자 응답 속도 개선**

- **복합 도메인/마이크로서비스 통합에 적합**
  - 여러 마이크로서비스가 동일 이벤트를 구독하면서, 각자의 데이터를 일관되게 유지

---

## 4. 단점과 어려움

- **복잡한 디버깅 & 추적(Observability)**
  - 요청 하나가 여러 이벤트와 서비스로 퍼져 나가기 때문에  
    “무슨 일이 어디서 잘못됐는지” 추적하기가 어렵다.
  - 분산 트레이싱, 로그 상관관계 ID(Correlation ID) 전략이 중요.

- **최종 일관성(Eventual Consistency)**
  - 이벤트 기반으로 데이터가 퍼지기 때문에, **즉시 일관성(Strong Consistency)** 를 유지하기 어렵고
  - 잠시 동안 서비스 간 데이터가 어긋날 수 있다.

- **이벤트 폭발과 설계 난이도**
  - 이벤트 종류가 많아지면 스키마/버전 관리, 흐름 이해가 어려워짐
  - “무엇을 이벤트로 정의할 것인가?”, “어디까지를 한 이벤트로 볼 것인가?” 설계가 중요

- **중복 처리, 재처리, 순서 보장 문제**
  - 네트워크/브로커 특성상, **이벤트 중복 소비** 또는 **순서 뒤바뀜(out-of-order)** 가능성
  - Idempotency(멱등성) 설계, 키 기반 파티셔닝, 재처리 전략 필요

## 5. 대표 패턴들

### 5.1 Pub/Sub 패턴

- 가장 기본적인 이벤트 드리븐 패턴
  - Producer는 토픽(Topic)에 이벤트를 발행
  - Consumer는 관심 있는 토픽을 구독하고 비동기 처리
- 예:
  - 주문 서비스 → `order.created` 이벤트 발행
  - 결제 서비스, 재고 서비스, 알림 서비스 등이 이 이벤트를 동시에 구독

### 5.2 이벤트 소싱(Event Sourcing)

- **현재 상태(State)를 직접 저장하는 대신, 상태를 만든 “이벤트의 흐름”을 저장**하는 패턴
  - 예: 계좌 잔액을 “숫자 하나”로 저장하는 대신,
    `AccountCreated`, `MoneyDeposited`, `MoneyWithdrawn` 등의 이벤트 시퀀스로 저장
- 장점:
  - 고급 감사 로그(Audit), 시간 여행(Time Travel), 재연(Replay)이 가능
  - 도메인 이벤트가 극도로 명시적
- 단점:
  - 개념/구현 난이도가 높고, 기존 CRUD 사고방식과 다르다.
  - 조회 모델(Query Model)을 따로 구성해야 하는 경우가 많음 (CQRS와 자주 같이 사용)

### 5.3 CQRS(Command Query Responsibility Segregation)

- **명령(Command: 상태 변경)과 조회(Query: 읽기)를 분리**하는 패턴
  - 쓰기 모델과 읽기 모델을 분리해 성능/확장성을 높임
- 이벤트 드리븐과의 결합:
  - Command 처리 후 이벤트 발행 → 여러 Read 모델이 이 이벤트를 구독해  
    자신에게 최적화된 조회용 데이터를 구성
  - 예: 주문 서비스가 `OrderCreated` 이벤트를 발행하면,
    - “사용자별 주문 리스트” Read 모델
    - “상품별 판매 통계” Read 모델  
    등이 이벤트를 구독해 각각 자신의 DB를 업데이트

## 6. 이벤트 설계 시 고려사항

1. **이벤트 이름과 의미를 명확히**
   - `order_created`, `payment_completed` 같이 **“무슨 일이 일어났는지”**가 드러나게
   - “명령(Command)”가 아니라 “사실(Fact)”을 표현

2. **이벤트 스키마(필드) 설계**
   - 필수 필드, 선택 필드, 추후 확장 가능성을 고려
   - 버전 필드(`version`)를 두어 스키마 변경을 관리하기도 함

3. **Idempotency(멱등성)**
   - 같은 이벤트가 여러 번 처리되어도 최종 결과가 같도록 설계
   - 예: 이벤트 ID + 처리 이력 저장을 통해 “이미 처리한 이벤트는 건너뛰기”

4. **오류 및 재처리 전략**
   - Consumer가 이벤트를 처리하다 실패할 경우:
     - 재시도(Backoff, Dead Letter Queue)
     - 인적 개입을 위한 대기 큐

5. **관찰 가능성(Observability)**
   - 트레이싱 ID/Correlation ID를 이벤트에 포함
   - 로그, 메트릭, 분산 트레이싱 시스템(Zipkin, Jaeger 등)과 연계

---

## 7. 언제 이벤트 드리븐 아키텍처를 선택할까?

**적합한 경우**
- 여러 마이크로서비스가 느슨하게 협업해야 할 때
- 비동기 처리로 사용자 응답 속도를 높이고 싶을 때
- 알림/통계/로그/데이터 파이프라인 등 “파생 처리”가 많을 때
- 도메인 이벤트가 명확하고, 감사/추적/분석 요구가 높은 경우

**지양하거나 신중해야 할 경우**
- 시스템이 아직 작고, 팀의 복잡도 감당 능력이 부족할 때
- 강한 즉시 일관성이 최우선이고, 비즈니스적으로 허용 가능한 지연이 거의 없을 때
- 이벤트 브로커 운용, 모니터링, 재처리 전략을 충분히 설계/운영할 여력이 없을 때

---

## 8. 요약

- 이벤트 드리븐 아키텍처는 **“무슨 일이 일어났다”는 이벤트를 중심으로 시스템을 설계**하고,
  **Producer–Broker–Consumer 구조**로 느슨한 결합과 비동기 확장을 얻는 방식이다.
- 대신, **최종 일관성, 이벤트 설계/버전 관리, 관찰 가능성, 멱등성** 같은 복잡한 문제를 함께 가져온다.
- 작은 시스템에서는 과할 수 있지만,  
  도메인이 커지고 마이크로서비스가 늘어나기 시작할 때  
  “언제, 무엇을 이벤트로 표현할지”를 고민하기 시작하면 큰 도움이 된다.
