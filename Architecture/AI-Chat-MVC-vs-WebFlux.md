# AI Chat 부하테스트에서 Spring MVC vs WebFlux 비교

> 연결된 정리본:
> - [AI Chat에서 MVC와 WebFlux 선택](../../../TIL.wiki/ai-chat-mvc-vs-webflux.md)


> 주제: **AI Chat 부하테스트**  
> 질문: “부하테스트 대회에서 AI Chat 서버를 만든다면, Spring MVC와 WebFlux 중 무엇이 더 좋은 선택인가?”

---

## 1. 전제: AI Chat + 부하테스트 시나리오

AI Chat 서비스의 주요 특징:

- **LLM 호출**: 외부 LLM/AI API 호출, 응답 대기 시간이 길다(수 초까지).
- **DB I/O**: 대화 로그, 유저 정보 저장/조회(보통 I/O 바운드).
- **스트리밍 응답**: 토큰 단위 스트리밍(SSE/WebSocket)으로 긴 연결 유지.

즉, 전형적인 **고동시성 + I/O 바운드** 서비스이며,

- 많은 유저가 동시에 접속해서
- 각 요청이 상대적으로 오래 유지되는(스트리밍) 패턴이다.

이런 상황에서는 **스레드를 얼마나 효율적으로 쓰는지**가 성능과 직결된다.

---

## 2. Spring MVC 특징 (Servlet + 블로킹)

### 2.1 모델

- 요청당 스레드 1개(Thread-per-request) 구조(Tomcat 등 서블릿 컨테이너).
- 대부분의 스택(JPA/JDBC, 동기 HTTP 클라이언트 등)이 블로킹 전제로 설계.

### 2.2 장점

- 코드가 단순하고 팀이 익숙한 경우가 많다.
- 동기 모델이라 디버깅/모니터링/튜닝이 비교적 쉽다.
- 동기 LLM SDK, JPA/JDBC 등과 자연스럽게 잘 맞는다.

### 2.3 단점 (AI Chat 부하테스트 관점)

- 각 요청이 **긴 스트리밍 연결(SSE/WebSocket) + LLM 대기 시간**을 가지면,
  - 요청당 스레드가 그 시간 동안 계속 점유된다.
  - 동시 유저 수가 늘어날수록 스레드 수·컨텍스트 스위칭 비용이 급증.
- 동시 접속 수가 매우 큰 시나리오에서는,
  - 스레드풀 한계와 메모리 사용량이 병목이 되기 쉽다.

---

## 3. Spring WebFlux 특징 (리액티브 + 논블로킹)

### 3.1 모델

- Netty/Undertow 기반 논블로킹 I/O + 적은 수의 이벤트 루프 스레드.
- `Mono`/`Flux` 기반 비동기 스트림, 스트리밍 응답(SSE/WebSocket)에 최적화.

### 3.2 장점

- **동시 연결 수가 매우 많고, 각 요청의 I/O 대기 시간이 길 때** 유리.
- 적은 수의 스레드로 많은 SSE/WebSocket 연결을 유지 가능.
- 같은 하드웨어로 더 많은 동시 유저를 수용하거나,
  - 동일 동시성에서 CPU·메모리 사용량을 줄일 잠재력이 크다.

### 3.3 단점 / 주의점

- “끝까지 리액티브”가 아닐 경우 이점이 줄어든다.
  - 내부에서 JPA/JDBC, 동기 HTTP 클라이언트(RestTemplate, 동기 LLM SDK)를 쓰면,
    - 이벤트 루프가 막히거나, 별도 스레드풀로 offload 필요.
  - 결과적으로 복잡도만 증가하고 성능 이점이 거의 없을 수 있다.
- 리액티브 체인 설계/디버깅/모니터링 난이도가 높다.

---

## 4. AI Chat 부하테스트 관점 비교

### 4.1 LLM·DB 호출 방식에 따른 차이

- **엔드투엔드 논블로킹 가능 (이상적인 WebFlux 시나리오)**  
  - `WebClient` 기반 LLM 호출, R2DBC·Reactive Mongo 등 사용.
  - 장점:
    - 긴 LLM 대기 시간 동안 스레드가 놀지 않음.
    - 적은 스레드로 많은 동시 요청 처리 가능.
    - 부하테스트에서 동시 유저 수 / 처리량 / 자원 효율 지표 개선 기대.

- **LLM/DB가 대부분 동기·블로킹 (현실적인 MVC 친화 시나리오)**  
  - 동기 HTTP 클라이언트, JPA/JDBC, 동기 SDK 등.
  - 이 경우:
    - MVC와 WebFlux 차이가 줄어들거나,
    - WebFlux 쪽이 오히려 셋업/튜닝 미숙으로 성능이 나빠질 위험도 있다.

### 4.2 스트리밍(SSE/WebSocket) 연결 수

- **Spring MVC**
  - 요청당 스레드 1개가 스트리밍 동안 계속 붙잡혀 있음.
  - 동시 유저 수가 커질수록 Tomcat 스레드 수를 많이 늘려야 하고,
  - 메모리·컨텍스트 스위칭 비용이 증가 → 한계점이 명확하게 드러난다.

- **Spring WebFlux**
  - 논블로킹 I/O 기반으로 적은 스레드로도 많은 연결 유지 가능.
  - AI Chat처럼 토큰을 10~30초 동안 스트리밍하는 경우,
  - 동일 자원에서 더 많은 유저를 수용할 잠재력이 크다.

---

## 5. 언제 Spring MVC가 더 나은 선택인가

다음 조건에 여러 개 해당하면 MVC가 현실적이고 안전한 선택일 수 있다.

- 팀이 Spring MVC/서블릿, JPA/JDBC, 동기 HTTP에 이미 익숙하다.
- LLM SDK/DB 드라이버가 동기 위주이며, 리액티브로 바꿀 여유가 적다.
- 대회에서 “최고 성능”보다 “안정적인 동작 + 빠른 디버깅”이 더 중요하다.

전략:

- Spring MVC + SSE/WebSocket으로 스트리밍 구현.
- 대신 다음을 적극 튜닝:
  - Tomcat 스레드 풀(max-threads 등).
  - DB 커넥션 풀.
  - HTTP 타임아웃, Keep-Alive, 압축(gzip) 설정.
- 구현 난이도가 낮고, 문제 발생 시 대응 속도가 빠르다.

---

## 6. 언제 Spring WebFlux가 더 나은 선택인가

다음 조건에 가깝다면 WebFlux 선택이 성능 면에서 유리하다.

- LLM 호출을 `WebClient` 등 논블로킹 HTTP로 구현 가능하다.
- DB도 R2DBC/Reactive Mongo 등 논블로킹 드라이버로 구성 가능하다.
- 팀 내에 리액티브 경험이 있거나, 실험·튜닝에 시간을 투자할 수 있다.
- 대회 채점 기준에 **동시 연결 수, 지연 시간, 자원 효율** 등이 포함되어 있다.

전략:

- WebFlux + SSE/WebSocket으로 토큰 스트리밍 구현.
- 블로킹 작업은 최대한 제거하거나, 별도 스케줄러/스레드풀로 격리.
- 이벤트 루프 스레드 수, backpressure, 타임아웃 등을 세밀하게 튜닝.

---

## 7. 결론 요약 (의사결정 가이드)

질문을 두 개로 단순화할 수 있다.

1. **엔드투엔드로 대부분 논블로킹으로 가져갈 수 있는가?**
2. **팀이 리액티브(WebFlux) 설계·디버깅에 익숙하고, 튜닝할 시간이 있는가?**

- 둘 다 “Yes”에 가깝다면:
  - → **Spring WebFlux**  
    - 고동시성 + 긴 스트리밍 연결이 있는 AI Chat 부하테스트에서
    - 동시 유저 수와 자원 효율 측면에서 더 유리할 가능성이 크다.

- 하나라도 “No”에 가깝다면:
  - → **Spring MVC**  
    - 익숙한 동기 스택 위에서 SSE/WebSocket을 구현하고,
    - Tomcat/DB 튜닝에 집중하는 것이
    - 제한된 시간의 대회 환경에서는 더 안정적이다.

요약하면,

- **리액티브에 익숙하고, 끝까지 논블로킹 체인을 만들 자신이 있다면 WebFlux로 “최대 성능”을 노리고,**
- **그렇지 않다면 MVC로 “안정적인 성능 + 빠른 대응”을 노리는 것이 현실적인 전략**이다.
