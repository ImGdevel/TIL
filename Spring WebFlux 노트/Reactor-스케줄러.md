# Reactor 스케줄러(Scheduler)

> 태그: `#webflux` `#reactor` `#scheduler` `#thread-model`<br>
> 작성일: 2026-06-25<br>
> 최종 수정일: 2026-06-25

## 정의

Reactor 스케줄러는 리액티브 체인의 각 단계가 어떤 스레드 풀에서 실행될지를 결정하는 추상화로, `subscribeOn`/`publishOn` 연산자와 결합해 I/O-bound와 CPU-bound 작업에 서로 다른 스레드 모델을 적용할 수 있게 한다.

## 특징 / 상세

### 1. 스케줄러 개요

- Reactor의 연산자 체인은 기본적으로 호출 스레드 위에서 동기적으로 실행되지만, 스케줄러를 지정하면 체인의 특정 구간을 다른 스레드(풀)로 옮겨 실행할 수 있다.
- 어떤 스케줄러를 선택하느냐에 따라 동시성 처리량, 블로킹 작업의 안전성, 리소스 사용량이 크게 달라진다.

### 2. 스케줄러 유형

#### 2.1 immediate

- 스레드 모델: 별도 스레드 전환 없이 현재 호출 스레드에서 즉시 실행.
- 특징: 가장 가벼우며, 스레드 전환 오버헤드가 전혀 없다.
- 사용 사례: 스레드 전환이 필요 없는 단순 동기 연산을 명시적으로 표시하고 싶을 때.

#### 2.2 single / newSingle

- 스레드 모델: 단일 스레드 하나를 재사용(`single`) 하거나 새로 생성(`newSingle`).
- 특징: 순서 보장이 중요한 작업에 적합하지만, 동시 처리량은 1개로 제한된다.
- 사용 사례: 순차적으로만 처리해야 하는 경량 작업.

#### 2.3 elastic (Deprecated)

- 스레드 모델: 필요에 따라 스레드를 무제한으로 늘리는 캐시드 스레드 풀.
- 특징: 스레드 수 제한이 없어 과도한 블로킹 작업이 몰릴 경우 시스템 자원을 무한히 소모할 위험이 있었다.
- 현재 상태: Deprecated — `boundedElastic`으로 대체되었다. 새 코드에서는 사용하지 않아야 한다.

#### 2.4 boundedElastic

- 스레드 모델: 필요에 따라 스레드를 늘리되, 상한(기본값은 CPU 코어 수 × 10, 추가로 대기열 한도 존재)이 있는 캐시드 스레드 풀.
- 특징: `elastic`의 무제한 확장 위험을 제거하면서도 블로킹 I/O 작업을 리액티브 체인에서 안전하게 격리할 수 있는 기본 권장 스케줄러.
- 사용 사례: JDBC, 파일 I/O, 블로킹 라이브러리 호출처럼 어쩔 수 없이 블로킹 코드를 실행해야 하는 구간.

#### 2.5 newBoundedElastic

- 스레드 모델: `boundedElastic`과 동일한 정책을 가진 새로운 전용 인스턴스를 생성.
- 특징: 전역 공유 `boundedElastic` 풀과 분리된, 특정 용도 전용 풀이 필요할 때 사용.
- 사용 사례: 특정 외부 시스템 호출만을 위한 격리된 스레드 풀을 구성하고 싶을 때.

#### 2.6 parallel

- 스레드 모델: CPU 코어 수에 맞춰 고정된 크기의 스레드 풀.
- 특징: CPU 집약적인 연산(계산, 직렬화/역직렬화 등)에 최적화되어 있으며, 스레드 수가 코어 수로 제한되어 컨텍스트 스위칭 오버헤드를 줄인다.
- 사용 사례: CPU-bound 연산(복잡한 변환, 암호화 연산 등).

#### 2.7 fromExecutorService

- 스레드 모델: 사용자가 직접 구성한 `ExecutorService`를 그대로 스케줄러로 감싸 사용.
- 특징: 기존에 운영 중인 커스텀 스레드 풀 정책(크기, 큐 정책 등)을 그대로 재사용할 수 있다.
- 사용 사례: 이미 애플리케이션 전반에서 표준화된 `ExecutorService`가 있고, 그 정책을 리액티브 체인에도 일관되게 적용하고 싶을 때.

### 3. I/O-bound vs CPU-bound

#### 3.1 I/O-bound 작업

- 특징: 네트워크 호출, 디스크 I/O처럼 스레드가 대부분의 시간을 "대기"하는 작업.
- 권장 스케줄러: `boundedElastic` — 대기 시간 동안 다른 작업이 스레드를 점유하지 않도록 스레드 수를 유연하게 늘릴 수 있다.

#### 3.2 CPU-bound 작업

- 특징: 실제 연산에 CPU를 지속적으로 사용하는 작업.
- 권장 스케줄러: `parallel` — 코어 수만큼의 스레드로 제한해, 코어 수를 초과하는 스레드 생성으로 인한 컨텍스트 스위칭 비용을 피한다.

### 4. 스레드 제어 연산자

#### 4.1 subscribeOn

- 구독이 발생하는 시점, 즉 업스트림(데이터 생성 쪽)이 실행되는 스레드를 지정한다.
- 체인 내 어디에 위치하든, 업스트림 전체의 실행 스레드에 영향을 준다(가장 위쪽 Publisher의 동작 스레드를 결정).

#### 4.2 publishOn

- 호출된 지점 이후의 다운스트림 연산자들이 실행되는 스레드를 변경한다.
- `subscribeOn`과 달리, 체인에서 호출된 위치 기준으로 그 이후 구간에만 영향을 준다 — 체인 내에서 여러 번 사용해 구간별로 다른 스레드를 적용할 수 있다.

### 5. 실용적 고려사항

#### 5.1 테스트 작성 시 주의점

- 스케줄러로 다른 스레드에서 비동기로 실행되는 코드는, 테스트 메인 스레드가 결과를 기다리지 않으면 검증 전에 테스트가 끝나버릴 수 있다.
- `StepVerifier`의 `verifyComplete()`나 `CountDownLatch` 등으로 비동기 작업의 완료를 명시적으로 기다려야 한다.

#### 5.2 정리

- 스케줄러 선택은 단순한 성능 튜닝이 아니라, 블로킹 코드를 안전하게 격리하고 시스템 자원(스레드 수)을 예측 가능하게 통제하는 핵심 설계 결정이다.
- I/O-bound에는 `boundedElastic`, CPU-bound에는 `parallel`을 기본 선택으로 삼고, 필요한 경우에만 `fromExecutorService`나 `newBoundedElastic`처럼 별도 격리된 풀을 구성한다.

## 트레이드오프

**규모 가정**: 단일 인스턴스, 가용 CPU 코어 수 N개, I/O 호출(DB/외부 API)과 CPU 연산이 혼재된 일반적인 WebFlux 서비스. 가정이 다르면(예: 순수 CPU 연산만 있는 배치성 서비스) 선택이 달라진다.

| 스케줄러 | 일관성 | 가용성 | 지연 | 비용(스레드/메모리) | 운영 부담 |
| --- | --- | --- | --- | --- | --- |
| `parallel` (CPU-bound용) | 영향 없음 | 코어 수 고정이라 과부하 시 큐잉 발생 가능 | 컨텍스트 스위칭 최소화로 낮음 | 코어 수만큼 고정, 예측 가능 | 낮음(고정 크기) |
| `boundedElastic` (I/O-bound용) | 영향 없음 | 상한 내에서 유연한 확장으로 가용성 높음 | I/O 대기 동안 다른 작업 처리 가능해 전체 처리율 향상 | 상한(코어×10 + 큐)까지 증가 가능 | 상한 도달 시 거부/대기 정책 모니터링 필요 |
| `elastic` (Deprecated) | 영향 없음 | 무제한 확장으로 일시적 가용성은 높아 보이나 과부하 시 전체 장애 위험 | 낮음(스레드 부족 없음) | 무제한 — 통제 불가능한 비용 | 매우 높음(사실상 사용 금지) |
| `fromExecutorService` (커스텀) | 영향 없음 | 커스텀 정책에 의존 | 커스텀 정책에 의존 | 커스텀 정책에 의존 | 기존 운영 정책과 일관성 유지 부담 |

**추천**: I/O-bound 구간은 `boundedElastic`, CPU-bound 구간은 `parallel`을 기본값으로 삼는다. `elastic`은 Deprecated이므로 신규/기존 코드 모두 `boundedElastic`으로 교체한다. 기존에 표준화된 `ExecutorService` 운영 정책(모니터링, 알람 등)이 이미 갖춰져 있는 조직이라면 `fromExecutorService`로 일관성을 유지하는 것도 합리적인 대안이다.
