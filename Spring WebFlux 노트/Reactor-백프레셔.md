# Reactor 백프레셔(Backpressure)

> 태그: `#webflux` `#reactor` `#backpressure`<br>
> 작성일: 2026-06-25<br>
> 최종 수정일: 2026-06-25

## 정의

백프레셔는 생산자(Publisher)가 소비자(Subscriber)의 처리 속도를 초과해 데이터를 발행하지 못하도록, 소비자가 `request(n)`으로 요청량을 직접 제어하는 리액티브 스트림즈의 흐름 제어 메커니즘이다.

## 특징 / 상세

### 1. 백프레셔가 필요한 이유

#### 1.1 생산자-소비자 속도 불균형

- 생산자가 소비자보다 빠르게 데이터를 만들어내면, 처리되지 못한 데이터가 쌓여 메모리 사용량이 급증하고 GC 압박, 응답 지연, 최악의 경우 OOM(OutOfMemoryError)으로 이어질 수 있다.
- 백프레셔는 이 불균형을 소비자 쪽에서 능동적으로 통제하는 장치다.

#### 1.2 기존 동기/블로킹 모델과 비교

- 전통적인 동기/블로킹 모델에서는 호출 스레드가 결과를 받을 때까지 블로킹되므로 생산자가 소비자를 앞서나갈 수 없다 (자연히 속도가 동기화됨).
- 비동기/논블로킹 모델에서는 이 자연스러운 동기화가 사라지므로, 별도의 명시적 흐름 제어 수단(백프레셔)이 반드시 필요하다.

### 2. 리액티브 스트림즈 명세에서의 백프레셔

#### 2.1 핵심 인터페이스 복습

- `Subscription.request(n)` — 소비자가 처리 가능한 만큼만 요청
- `Subscription.cancel()` — 더 이상 데이터가 필요 없을 때 구독 해지

#### 2.2 프로토콜 흐름

1. 구독 (`subscribe`)
2. 구독 정보 전달 (`onSubscribe`)
3. 수요 요청 (`request(n)`)
4. 요청량만큼 데이터 전달 (`onNext` × n)
5. 종료 (`onComplete` / `onError`)

이 흐름에서 핵심은 3번이다. Publisher는 Subscriber가 `request`로 명시한 개수 이상은 절대 보내지 않아야 한다는 것이 명세의 강제 규약이다.

### 3. Reactor에서의 백프레셔 기본 동작

#### 3.1 기본 구현

- `subscribe()`를 인자 없이 호출하면 내부적으로 무제한 요청(`Long.MAX_VALUE`)을 보내는 기본 Subscriber가 생성되어, 별도 설정 없이는 백프레셔가 사실상 비활성화된 것처럼 동작한다.
- 명시적으로 흐름을 제어하려면 `BaseSubscriber`를 상속해 `request()` 호출 시점과 양을 직접 결정해야 한다.

#### 3.2 콜드/핫 퍼블리셔와 백프레셔

- 콜드 퍼블리셔: 구독자별로 독립적인 스트림이 생성되므로, 각 구독자가 자신의 속도에 맞춰 백프레셔를 적용할 수 있다.
- 핫 퍼블리셔: 데이터 생성이 구독 여부와 무관하게 진행되는 경우가 많아, 느린 구독자가 `request`로 속도를 늦춰도 생산 자체를 멈추지 못할 수 있다 — 이때 `onBackpressureXXX` 연산자로 별도 완충 전략이 필요해진다.

### 4. BaseSubscriber를 이용한 수요 직접 제어

#### 4.1 소개

- `hookOnSubscribe`에서 최초 요청량을 정하고, `hookOnNext`에서 처리 후 추가 요청을 보내는 방식으로 수요를 직접 제어한다.

#### 4.2 단계적 요청 예시

- 한 번에 1개씩 요청 → 처리 → 완료 후 다시 1개 요청을 반복하면, 생산자는 소비자가 받을 준비가 됐다고 신호를 보낸 만큼만 데이터를 내보낸다.

#### 4.3 동적 요청량 조절

- 시스템 부하(CPU, 큐 길이 등)를 모니터링해 요청량을 늘리거나 줄이는 방식으로, 고정된 배치 크기보다 더 정교한 흐름 제어가 가능하다.

### 5. onBackpressure 연산자

#### 5.1 공통 전제

- `onBackpressureXXX` 연산자들은 다운스트림의 요청 속도를 업스트림이 따라가지 못할 때(즉 오버플로우 상황) 어떻게 대응할지를 정의한다.

#### 5.2 onBackpressureBuffer

- 처리되지 못한 데이터를 버퍼에 쌓아두고 소비자가 준비되면 순서대로 전달한다.
- 버퍼 크기를 제한하지 않으면 메모리 사용량이 무한히 늘어날 위험이 있다.

#### 5.3 onBackpressureDrop

- 소비자가 받을 준비가 안 된 데이터는 버리고, 이후 들어오는 새로운 데이터만 전달한다.
- 데이터 유실을 허용하되 메모리 사용량을 일정하게 유지하려는 경우에 적합하다.

#### 5.4 onBackpressureLatest

- 가장 최신 데이터만 유지하고 이전 데이터는 버린다.
- "최신 상태"만 의미 있는 경우(예: 센서 값, 실시간 상태 업데이트)에 적합하다.

### 6. 고수준 연산자와 백프레셔의 관계

#### 6.1 buffer

- 일정 개수/시간 단위로 데이터를 묶어 다운스트림에 전달함으로써, 다운스트림 호출 빈도를 줄이고 처리 부담을 완충한다.

#### 6.2 take

- 처리할 데이터의 총량 자체를 제한해, 무한 스트림에서도 소비자가 부담해야 할 전체 작업량을 통제한다.

#### 6.3 limitRate

- 업스트림에 보내는 요청을 작은 단위로 나누어 prefetch함으로써, 한 번에 너무 많은 데이터가 몰리는 것을 방지한다.

### 7. 스레드/스케줄러와 백프레셔

#### 7.1 publishOn / subscribeOn

- 백프레셔 신호와 실제 데이터 처리가 어느 스레드에서 일어나는지는 `publishOn`/`subscribeOn` 배치에 따라 달라지며, 스케줄러 선택이 처리 속도(따라서 체감 백프레셔 압박)에 직접 영향을 준다.

#### 7.2 Bounded Elastic

- I/O 바운드 작업에 적합한 `boundedElastic` 스케줄러는 스레드 풀 크기에 상한이 있어, 무분별하게 스레드를 늘려 백프레셔 문제를 "스레드로 덮어버리는" 것을 방지한다.

### 8. 시스템 아키텍처 관점의 백프레셔

#### 8.1 메시지 큐와 토픽

- 애플리케이션 레벨의 백프레셔만으로 부족할 때, Kafka 등의 메시지 큐를 앞단에 두어 생산과 소비 속도를 시스템 아키텍처 수준에서 분리할 수 있다.

#### 8.2 외부 시스템 보호

- 백프레셔는 외부 시스템(DB, 외부 API)을 과도한 요청으로부터 보호하는 역할도 한다. 서킷 브레이커(Resilience4j 등)와 결합하면, 백프레셔로 속도를 늦추는 것과 더불어 임계치 초과 시 요청을 차단하는 이중 방어선을 구성할 수 있다.

### 9. 자주 하는 실수

1. `subscribe()`를 인자 없이 호출해 백프레셔가 비활성화된 것을 인지하지 못함.
2. `onBackpressureBuffer`에 크기 제한을 두지 않아 메모리 누수로 이어짐.
3. 핫 퍼블리셔의 생산 속도 자체는 줄일 수 없다는 점을 간과하고, 구독자 쪽 제어만으로 모든 문제를 해결하려 함.
4. 블로킹 코드를 리액티브 체인 안에 그대로 섞어 써서, 스케줄러의 스레드를 점유시켜 버리고 백프레셔가 무력화되는 상황을 만듦.

### 10. 정리

- 백프레셔는 리액티브 스트림즈의 핵심 규약이며, Reactor에서는 기본 `subscribe()`가 무제한 요청을 보낸다는 점을 인지하고 필요한 경우 `BaseSubscriber`나 `onBackpressureXXX` 연산자로 명시적 제어를 추가해야 한다.
- 콜드/핫 퍼블리셔, 스케줄러 선택, 상위 아키텍처(메시지 큐, 서킷 브레이커)까지 함께 고려해야 실효성 있는 백프레셔 전략이 완성된다.

## 트레이드오프

**규모 가정**: 단일 인스턴스 기준 초당 수백~수천 건의 이벤트(QPS) 스트림을 처리하는 WebFlux 애플리케이션. 가정이 다르면(예: QPS가 매우 낮아 생산/소비 속도 불균형이 실질적으로 발생하지 않는 경우) 아래 비교의 실익이 사라진다.

| 전략 | 일관성 | 가용성 | 지연 | 비용(메모리) | 운영 부담 |
| --- | --- | --- | --- | --- | --- |
| `onBackpressureBuffer` | 데이터 유실 없음(높은 일관성) | 버퍼 가득 차면 OOM으로 전체 가용성 저하 위험 | 버퍼 대기로 지연 증가 가능 | 버퍼 크기에 비례해 증가, 무제한 시 위험 | 버퍼 크기/오버플로우 정책 모니터링 필요 |
| `onBackpressureDrop` | 데이터 유실 허용(낮은 일관성) | 메모리 고갈 위험 없어 가용성 높음 | 유실분 재처리 없어 지연 낮음 | 일정하게 낮음 | 유실률 모니터링 필요 |
| `onBackpressureLatest` | 최신 값만 보장(부분적 일관성) | 가용성 높음 | 낮음 | 일정하게 낮음(슬롯 1개) | 비즈니스상 "최신 값만 의미 있는지" 사전 확인 필요 |
| 메시지 큐(Kafka 등) 도입 | 큐의 영속성에 따라 높은 일관성 가능 | 큐 자체가 SPOF가 되지 않도록 별도 가용성 설계 필요 | 큐 적재/소비 단계 추가로 지연 증가 | 큐 운영/스토리지 비용 추가 | 큐 클러스터 운영 부담 추가 |

**추천**: 데이터 유실이 비즈니스적으로 허용되지 않는 경우(결제, 주문 등)는 `onBackpressureBuffer` + 명시적 버퍼 크기 제한 + 모니터링을 기본으로 하고, 유실 허용 가능한 실시간 스트림(센서, 로그, 메트릭)은 `onBackpressureDrop`/`onBackpressureLatest`를 우선 검토한다. 애플리케이션 레벨 백프레셔만으로 부족한 대용량/다중 인스턴스 환경은 Kafka 등 메시지 큐를 앞단에 두는 아키텍처 레벨 해법으로 확장한다.
