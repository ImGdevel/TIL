# Reactor 예외 처리

> 태그: `#webflux` `#reactor` `#error-handling`<br>
> 작성일: 2026-06-25<br>
> 최종 수정일: 2026-06-25

## 정의

Reactor에서 예외는 스트림 중단 후 `onError` 신호를 통해 구독자에게 전파되며, `onErrorReturn`/`onErrorResume`/`onErrorContinue`/`onErrorMap`/`onErrorStop`/`retry` 등의 연산자로 신호가 발생한 시점에 어떤 복구·전환·중단 전략을 취할지 선택적으로 제어할 수 있다.

## 특징 / 상세

### 1. 기본 에러 처리 흐름

Reactor 스트림에서 예외가 발생하면 다음 6단계를 거친다.

1. 예외 캡처 및 종료 — 체인 내부에서 예외가 발생하면 해당 단계에서 정상적인 값 발행이 즉시 중단된다.
2. 메시지 발행 중단 — 예외 발생 이후로는 더 이상 `onNext`가 호출되지 않는다.
3. 후속 처리 미진행 — 예외 발생 지점 이후의 연산자들은 실행되지 않는다.
4. `onError` 전파 — 예외가 `onError(Throwable)` 신호로 변환되어 다운스트림으로 전파된다.
5. 구독자 수신 — 최종 구독자(또는 중간의 에러 처리 연산자)가 `onError` 신호를 받는다.
6. `onComplete` 미호출 — 정상 종료 신호인 `onComplete`는 호출되지 않는다 — 즉, 에러로 끝난 스트림은 "성공적으로 끝났다"는 신호를 받지 않는다.

### 2. 에러 처리 연산자 분석

다음 예시는 모든 연산자에서 공통으로 사용하는 시나리오다: `Flux.just(1, 2, 0, 3)`에서 각 값으로 어떤 수를 나누다가 `0`에서 `ArithmeticException`이 발생하는 상황.

#### 2.1 onErrorReturn

- 목적: 예외 발생 시 미리 정의한 대체 값 하나로 대체하고 스트림을 정상 종료시킨다.
- 동작 방식: 예외가 감지되면 `onNext(대체값)`을 호출한 뒤 즉시 `onComplete`를 호출한다. 이후의 원래 데이터(`3`)는 더 이상 방출되지 않는다.
- 예시 시나리오: `1, 2`까지 정상 방출 → `0`에서 예외 발생 → 대체 값(예: `-1`) 방출 → 완료. `3`은 유실된다.

#### 2.2 onErrorResume

- 목적: 예외 발생 시 대체 Publisher(`Mono`/`Flux`)로 전환해 스트림을 이어간다.
- 동작 방식: 예외가 감지되면 지정한 대체 Publisher를 구독하여 그 결과를 이어서 방출한다 — 폴백 로직(캐시 조회, 기본값 시퀀스 등)을 자연스럽게 연결할 수 있다.
- 예시 시나리오: `0`에서 예외 발생 → `Flux.just(-1, -2)` 같은 대체 시퀀스로 전환되어 이어서 방출 후 완료.

#### 2.3 onErrorContinue

- 목적: 예외가 발생한 해당 요소만 건너뛰고, 나머지 요소는 계속 처리한다.
- 동작 방식: 예외를 발생시킨 값과 예외 객체를 콜백으로 받아 로깅 등의 처리를 한 뒤, 스트림 자체는 종료되지 않고 다음 요소로 계속 진행한다.
- 예시 시나리오: `1, 2`까지 정상 방출 → `0`에서 예외 발생, 해당 요소만 건너뜀(로그 등 부가 처리) → `3` 정상 방출 → 완료.
- 주의: 모든 연산자가 `onErrorContinue`를 지원하지는 않으며, 연산자 체인 중간에 사용 시 예상과 다르게 동작할 수 있어 Reactor 공식 문서에서도 신중한 사용을 권고한다.

#### 2.4 onErrorMap

- 목적: 발생한 예외를 다른 타입의 예외로 변환해 전파한다.
- 동작 방식: 원본 예외(`ArithmeticException`)를 받아 애플리케이션에 맞는 커스텀 예외(예: `BusinessException`)로 매핑한 뒤, 변환된 예외로 `onError`를 호출한다. 스트림은 여전히 에러로 종료된다.
- 예시 시나리오: `0`에서 `ArithmeticException` 발생 → `CalculationFailedException`으로 변환되어 구독자에게 전파.

#### 2.5 onErrorStop

- 목적: 기본 동작(에러 발생 시 즉시 중단)을 명시적으로 보장한다 — 주로 `onErrorContinue`가 적용된 체인의 특정 구간에서 다시 "예외 발생 시 즉시 중단" 동작으로 되돌리기 위해 사용한다.
- 동작 방식: 해당 연산자 이후로는 `onErrorContinue`의 효과가 적용되지 않고, 표준 흐름(예외 발생 즉시 종료)으로 복귀한다.

#### 2.6 retry

- 목적: 예외 발생 시 업스트림을 재구독하여 처음부터(또는 지정한 시점부터) 다시 시도한다.
- 동작 방식: 지정한 횟수만큼, 혹은 지정한 조건(`retryWhen`을 사용한 백오프 전략 등)에 따라 재시도한다. 모든 재시도가 실패하면 최종적으로 `onError`가 전파된다.
- 예시 시나리오: `0`에서 예외 발생 → 전체 시퀀스를 처음부터 재구독 → 동일한 입력이라면 동일한 지점에서 다시 예외 발생 → 횟수 초과 시 최종 에러 전파.
- 주의: 원본 시퀀스가 부작용(side effect)이 있는 연산(예: 외부 API 호출, DB 쓰기)을 포함하면, 재시도 시 그 부작용이 중복 실행될 위험이 있다.

### 3. 글로벌 에러 핸들링

- Reactor는 `Hooks.onErrorDropped` 등 전역 훅(Hooks)을 통해 어디서도 처리되지 않은 예외나 의도적으로 무시된 에러 신호를 한곳에서 감지/로깅할 수 있다.
- Spring MVC의 `@RestControllerAdvice`가 컨트롤러 레벨에서 예외를 한곳에 모아 처리하는 것과 유사하게, 리액티브 체인에서도 전역 처리 지점을 둘 수 있다는 점에서 개념적으로 대응된다.

### 4. 개발자 관점에서의 고려사항

- 어떤 연산자를 선택할지는 "예외 상황에서 무엇을 우선할 것인가"에 따라 달라진다 — 가용성(어떤 값이든 응답)을 우선할지, 일관성(잘못된 값을 보내지 않고 명확히 실패를 알릴지)을 우선할지가 핵심 판단 기준이다.
- 부작용이 있는 작업에는 `retry`를 신중하게 적용해야 하며, 멱등성(idempotency)이 보장되지 않은 작업의 재시도는 데이터 중복/오염을 유발할 수 있다.
- `onErrorContinue`는 강력하지만 모든 연산자에서 지원되지 않고 부작용도 있어, 공식 문서에서도 "최후의 수단"으로 권고되는 경향이 있다 — 사용 전 최신 Reactor 문서로 지원 여부와 권고사항을 확인해야 한다.

### 5. 정리

- Reactor의 에러 처리는 "예외 발생 → 스트림 중단 → onError 전파"라는 기본 골격 위에, 어디서 어떻게 복구/전환/중단할지를 연산자 선택으로 세밀하게 제어하는 구조다.
- `onErrorReturn`(대체값), `onErrorResume`(대체 스트림), `onErrorContinue`(요소 단위 스킵), `onErrorMap`(예외 변환), `onErrorStop`(기본 동작 복귀), `retry`(재시도)는 각각 다른 트레이드오프를 가지므로 상황에 맞게 선택해야 한다.

## 트레이드오프

**규모 가정**: 외부 API/DB 호출을 포함한 단일 WebFlux 서비스의 요청 처리 체인. 부작용이 있는 호출(결제, 쓰기 작업)이 포함되어 있는지에 따라 적합한 전략이 달라진다.

| 전략 | 일관성 | 가용성 | 지연 | 비용(운영) | 운영 부담 |
| --- | --- | --- | --- | --- | --- |
| `onErrorReturn` | 낮음(오류를 대체값으로 은폐) | 높음(항상 응답 반환) | 낮음(즉시 대체값) | 낮음 | 대체값이 실제 정상값과 혼동되지 않도록 모니터링 필요 |
| `onErrorResume` | 중간(폴백 로직 품질에 의존) | 높음(폴백 경로로 응답 지속) | 폴백 호출 비용만큼 증가 | 폴백 인프라(캐시 등) 필요 시 증가 | 폴백 경로도 별도 장애 대상이 됨 |
| `onErrorContinue` | 중간(일부 요소 누락 허용) | 높음(나머지 요소는 계속 처리) | 낮음 | 낮음 | 연산자별 지원 여부 확인 필요, 누락 데이터 추적 필요 |
| `onErrorMap` | 영향 없음(단순 예외 변환) | 영향 없음 | 영향 없음 | 낮음 | 예외 타입 매핑 규칙 유지보수 |
| `retry` | 높음(성공 시 원래 결과 보장) | 재시도 중 지연·부하 증가로 가용성 저하 가능 | 재시도 횟수만큼 증가 | 재시도로 인한 외부 호출 비용 증가 | 멱등성 미보장 시 중복 처리 위험, 백오프 전략 필요 |

**추천**: 외부 의존성 호출에는 기본적으로 `onErrorResume` + 명확한 폴백을 우선 적용하고, 멱등성이 보장된 호출에 한해 지수 백오프를 적용한 `retryWhen`을 추가로 결합한다. 단순 항목 단위 스트림 처리(배치성 변환)에서 일부 실패를 허용할 수 있다면 `onErrorContinue`를 검토하되, 사용 중인 연산자가 이를 지원하는지 최신 문서로 먼저 확인한다. 부작용이 있는 쓰기 작업에는 멱등성 보장 없이 `retry`를 적용하지 않는다.
