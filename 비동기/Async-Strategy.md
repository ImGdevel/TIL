# 비동기(Async) 전략 정리

> 연결된 정리본:
> - [비동기 전략과 쓰레드 풀 관리](../../../TIL.wiki/async-strategy-and-thread-pool-management.md)


> “당장 응답해야 하는 일과, 나중에 따로 처리해도 되는 일을 어떻게 나눌 것인가?”

---

## 1. 동기 vs 비동기, 한 줄 정리

- **동기(Sync)**  
  - 호출한 쪽이 작업이 끝날 때까지 기다린다. (요청 → 처리 → 응답)
- **비동기(Async)**  
  - 호출한 쪽은 “작업을 요청했다”는 사실만 확인하고 **결과는 나중에** 받거나, 아예 신경 쓰지 않는다.

비동기를 고려하는 이유:

- 요청 응답 시간을 줄이고 싶다.  
- 외부 시스템/느린 작업 때문에 전체 서비스가 느려지는 걸 막고 싶다.  
- 피크 트래픽을 완충하고, 뒤에서 천천히 처리하고 싶다.

---

## 2. 비동기로 풀고 싶은 문제 유형

1. **사용자 응답은 빨리 주고 싶은데, 뒤에서 할 일이 많을 때**
   - 예: 주문 완료 응답은 바로 주고, 포인트 적립/메일 발송/알림은 나중에 처리

2. **외부 API가 느리거나 불안정할 때**
   - 예: 제3자 결제 연동, 슬랙/메일/SMS 연동

3. **단기간에 요청이 몰리는 피크가 있을 때**
   - 예: 이벤트 오픈, 티켓팅, 정기 배치가 겹치는 시간

4. **주기적/배치성 작업**
   - 예: 매 5분마다 집계, 하루 한 번 리포트 생성

---

## 3. 비동기 전략의 레벨 나누기

비동기 전략은 대략 세 레벨로 나눌 수 있다.

1. **프로세스 내부 비동기 (In-Process Async)**
   - 쓰레드 풀, `CompletableFuture`, `@Async`, Reactor(WebFlux) 등
   - 한 애플리케이션 프로세스 안에서 동시성/비동기 처리

2. **서비스 간 비동기 (Cross-Service Async)**
   - MQ, Kafka, Redis Stream, Pub/Sub, Webhook 등
   - 서로 다른 서비스/프로세스 간의 비동기 메시징

3. **클라이언트-서버 비동기**
   - HTTP 요청 후 폴링, 콜백 URL, SSE/WebSocket, Push 알림 등
   - “사용자가 결과를 언제/어떻게 확인할 것인지”에 관한 전략

어떤 레벨에서 비동기를 쓸지부터 정하면, 쓸 도구가 자연스럽게 좁혀진다.

---

## 4. 프로세스 내부 비동기 – 쓰레드와 @Async

### 4-1. 쓰레드 풀 기반 비동기

대표적인 도구:

- `ExecutorService`, `ThreadPoolExecutor`
- Spring의 `ThreadPoolTaskExecutor` + `@Async`
- `CompletableFuture`, parallel stream 등

사용 예:

- 독립적인 I/O 호출 여러 개를 병렬로 보내고 싶을 때  
- 비즈니스 로직 일부를 “백그라운드 작업”으로 돌리고 싶을 때

주의할 점:

- 같은 JVM 안의 작업이기 때문에, **장애가 나면 같이 죽는다.**  
- 쓰레드 풀 크기/큐 설정을 잘못 잡으면 오히려 더 느려질 수 있다. (`ThreadPool-Management.md` 참고)
- `@Async`/AOP 기반 비동기는 self-invocation 이슈(같은 클래스 내부 호출 시 비동기 미적용)를 주의.

### 4-2. Non-Blocking/Reactive (WebFlux 등)

- 유저 수가 아주 많고, 대부분 I/O 대기인 시스템에서는  
  non-blocking I/O + 이벤트 루프 기반 구조가 더 적합할 수 있다.
- 대신 코드/디버깅 난이도가 올라가고, 팀 전체가 리액티브 패러다임을 이해해야 한다는 부담이 있다.

---

## 5. 서비스 간 비동기 – MQ·Kafka·Redis 등

### 5-1. 메시지 큐(MQ)

- RabbitMQ, SQS, ActiveMQ 등
- “작업을 큐에 넣고, Worker가 하나씩 꺼내 처리”하는 모델
- 장점:
  - 요청 응답과 실제 작업 처리를 분리
  - 재시도/DLQ, 백프레셔 등을 브로커에서 지원
- 단점:
  - 시스템 구조가 복잡해지고, 트랜잭션/일관성 고민이 필요 (`메시지 큐(MQ)로 서비스 간 작업을 나눠보기.md` 참고)

### 5-2. 이벤트 로그/스트림 (Kafka, Redis Stream)

- 여러 서비스가 같은 이벤트를 각자 속도로 소비해야 할 때 적합
- “한 번 처리하고 끝”이라기보다, **여러 Consumer Group이 같은 이벤트를 여러 번 읽는 구조**
- 장점:
  - 재처리/리플레이, 분석/파이프라인에 활용 가능
- 단점:
  - 운영/모니터링 복잡도, 데이터 모델링 부담 증가

### 5-3. 가벼운 신호용 – Redis Pub/Sub

- 내구성이 꼭 필요하지 않은 실시간 신호/알림에는 Pub/Sub도 선택지
- “실시간 알림은 비동기로, 로그/정산은 Kafka/DB로”처럼 역할 분리 가능

## 6. 클라이언트-서버 비동기 – 결과를 어떻게 돌려줄까?

비동기로 작업을 넘긴 뒤, 클라이언트 입장에서 **결과를 어떻게 받게 할지**도 전략이다.

1. **Fire-and-Forget**
   - “성공 여부는 나중에 로그/관리 화면에서 확인”  
   - 예: 알림 발송, 로그 적재

2. **Polling (폴링)**
   - 클라이언트가 `jobId`를 받아서 주기적으로 상태를 조회  
   - 예: 긴 처리시간이 필요한 리포트 생성

3. **Callback/Webhook**
   - 클라이언트(또는 다른 서버)가 콜백 URL을 제공하고,  
   - 작업이 끝나면 서버가 그 URL로 결과를 POST

4. **SSE/WebSocket**
   - 서버가 작업 상태/결과를 실시간 스트리밍으로 보내줌 (`SSE-vs-WebSocket.md` 참고)

어떤 방식을 쓸지는:

- 결과를 꼭 사용자 화면에 즉시 보여줘야 하는지  
- 클라이언트가 서버를 계속 호출할 여유가 있는지  
- 양방향 실시간 연결을 유지해도 되는지

에 따라 달라진다.

## 7. 비동기 설계 시 공통으로 신경 쓸 것들

1. **일관성과 트랜잭션**
   - “DB에 저장 + 메시지 발행”이 분리되면, **한쪽만 성공하는 경우**가 생길 수 있다.
   - Outbox 패턴, Transactional Messaging 등을 검토해야 한다.

2. **중복 처리 (Idempotency)**
   - 대부분의 비동기 메시징은 at-least-once 전달을 전제로 한다.
   - 같은 작업이 여러 번 들어와도 안전하도록, Consumer/핸들러를 idempotent하게 설계해야 한다.

3. **순서 보장**
   - “무조건 순서대로 처리되어야 한다”는 요구는 비동기 설계를 크게 어렵게 만든다.
   - 정말 필요한 경우가 아니라면, 순서 의존성을 줄이거나 특정 키별(partition) 순서 보장 정도로 좁히는 게 좋다.

4. **타임아웃과 재시도**

   - 비동기는 “언젠가 완료될 것”을 전제로 하지만,  
     실제로는 실패/지연 케이스를 항상 고려해야 한다.
   - 재시도 횟수, backoff 전략, DLQ 등을 명확히 정의할 것.

5. **모니터링과 관찰성(Observability)**

   - 동기는 보통 한 요청 안에서 끝나지만,  
     비동기는 여러 시스템/쓰레드를 거치면서 흐름이 끊어진다.
   - Trace ID/Correlation ID를 유지하고, 로그/메트릭/트레이싱으로 끝까지 추적 가능하게 만들어야 한다.

---

## 8. 정리 – “무엇을 비동기로 보낼 것인가?”부터

비동기를 도입할 때 가장 먼저 할 질문은 이거다.

> “이 작업은 **지금 당장** 끝나야 하는가,  
>  아니면 어느 정도 지연/유실을 허용하고 나중에 처리해도 되는가?”

이 질문에 답을 하다 보면,

- 동기로 남겨야 할 핵심 트랜잭션과  
- 비동기로 보내도 되는 부가 작업이 자연스럽게 나뉜다.

그 위에서:

- 프로세스 내부 비동기 (@Async, 쓰레드 풀, 리액티브)  
- 서비스 간 비동기 (MQ, Kafka, Redis Pub/Sub/Stream)  
- 클라이언트-서버 비동기 (폴링, 콜백, SSE/WebSocket)

을 적절히 조합해 가는 게 “비동기 전략”이라고 정리할 수 있다.  
새 기능을 설계할 때마다, “이건 어디까지 동기로 가져가고, 어디부터 비동기로 넘길까?”를 습관처럼 물어보면 도움이 된다.
