## 정의

`Coroutine`은 실행을 "중간에 멈추고 나중에 이어서" 할 수 있는 함수 단위의 동시성 구성 요소이며, 그 자체로 CPU 병렬성을 보장하지는 않는다.

## 특징 / 상세

핵심 키워드: `suspend/resume`, `cooperative`, `scheduler`, `concurrency`, `parallelism`

코루틴은 스레드보다 가볍게 동시성을 표현할 수 있지만, 실제 병렬 실행은 여러 스레드/프로세스와 결합될 때 만들어진다.

### 1. Process / Thread / Coroutine 차이(감각)

> "누가 스케줄링하고, 무엇을 공유하나"가 핵심이다.

- Process: 주소 공간이 분리된다.
- Thread: 같은 프로세스 주소 공간을 공유한다.
- Coroutine: 보통 같은 스레드 안에서 "협력적(cooperative)"으로 실행 흐름을 교대한다.

### 2. 왜 코루틴을 쓰나

> 동시성을 "코드 흐름"으로 표현하기 쉬워진다.

- I/O 대기 같은 구간에서 스레드를 막지 않고 다른 작업으로 전환하기 좋다.
- 콜백 지옥을 줄이고, 직관적인 순차 코드 형태를 유지할 수 있다.

### 3. 중요한 오해 하나

> 코루틴은 멀티스레드가 아니다.

- 코루틴이 한 스레드에서만 돌면, 동시에 "여러 코어에서" 실행되지 않는다.
- CPU 병렬성을 원하면 여러 스레드/프로세스(또는 스레드 풀)와 함께 사용해야 한다.

### 4. 자주 헷갈리는 포인트

> 코루틴은 "동시성 표현"에 강하고, 스레드는 "실제 실행 주체"에 가깝다.

- 코루틴은 보통 직접 선점되지 않고, `suspend/resume` 지점에서 흐름을 넘긴다.
- 그래서 긴 CPU 연산을 코루틴에만 맡기면, 동시성은 보여도 응답성은 나빠질 수 있다.

### 5. 실전 포인트

> I/O 중심 코드에서 코루틴의 장점이 가장 크게 드러난다.

- 네트워크 호출, 파일 I/O, DB 대기처럼 "기다림이 많은 작업"과 잘 맞는다.
- 반대로 CPU 바운드 작업은 코루틴보다 작업 분할, 스레드 풀, 병렬 처리 전략이 더 중요하다.
