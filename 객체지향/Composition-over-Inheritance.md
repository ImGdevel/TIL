# Composition over Inheritance

> 태그: `#oop` `#design-principle` `#coupling`<br>
> 작성일: 2026-06-25<br>
> 최종 수정일: 2026-06-25

## 정의

상속은 결합도가 높아지기 쉬우므로, "기능 확장"은 가능하면 상속보다 `composition`(구성/위임)으로 설계하는 편이 변경에 강하다.

## 특징 / 상세

핵심 키워드: `composition`, `delegation`, `coupling`, `is-a`, `has-a`

상속이 항상 나쁜 건 아니지만, 재사용 목적의 상속은 유지보수 비용을 키우기 쉽다.

### 1. 상속이 위험해지는 포인트

> 부모 구현에 강하게 묶여서, 부모 변경이 자식 동작을 깨뜨릴 수 있다.

- override/훅 메서드가 많아질수록 암묵적 규칙이 늘어난다.
- LSP(리스코프 치환) 위반이 쉽게 생긴다.

### 2. composition이 주는 이점

> 구현을 "가져다 쓰고(위임)" 교체할 수 있어 유연하다.

- 런타임에 구성 요소를 바꾸기 쉽다.
- 테스트에서 대체 구현을 넣기 쉽다.

### 3. 언제 상속이 맞나

> 진짜 "is-a" 관계이고, 공통 규약을 안정적으로 제공할 때다.

- 프레임워크의 확장 포인트처럼, 상속을 전제로 설계된 경우

### 4. 실전 포인트

> 상속은 "공통 코드 재사용"보다 "행동 교체" 관점에서 더 신중해야 한다.

- 구현이 바뀔 가능성이 높으면 합성이 안전하다.
- 상속은 API 계약과 부모 변화까지 같이 떠안는다는 점을 기억한다.
