# JVM 런타임 데이터 영역

> 태그: `#jvm` `#memory`<br>
> 작성일: 2026-06-23<br>
> 최종 수정일: 2026-06-23

## 정의

JVM은 실행 중 메모리를 Heap·Method Area(전체 스레드 공유)와 JVM Stack·PC Register·Native Method Stack(스레드별 독립)으로 나눈 5개 런타임 데이터 영역으로 관리한다.

## 특징 / 상세

| 영역 | 공유 범위 | 내용 | 발생 가능 예외 |
|---|---|---|---|
| Heap | 전체 스레드 공유 | 객체·배열 인스턴스. GC 대상. | OutOfMemoryError |
| Method Area (HotSpot 구현: Metaspace) | 전체 스레드 공유 | 클래스 구조, 메서드/필드 메타데이터, 메서드 코드, Run-Time Constant Pool | OutOfMemoryError |
| JVM Stack | 스레드별 | 프레임(지역변수 배열, 오퍼랜드 스택, 메서드 호출·리턴 정보) | StackOverflowError, OutOfMemoryError(동적 확장 실패 시) |
| PC Register | 스레드별 | 현재 실행 중인 바이트코드 명령어 주소. native 메서드 실행 중엔 값이 정의되지 않음 | 없음 |
| Native Method Stack | 스레드별 | JNI 등 네이티브(C) 코드 실행용 스택 | StackOverflowError, OutOfMemoryError(동적 확장 실패 시) |

핵심 포인트:

- Run-Time Constant Pool은 별도 영역이 아니라 Method Area에서 클래스/인터페이스 단위로 할당된다([JVMS §2.5.5](https://docs.oracle.com/javase/specs/jvms/se21/html/jvms-2.html#jvms-2.5.5)). 클래스가 로딩될 때 생성된다.
- JVMS는 "Method Area"라는 논리적 개념만 정의하고 구현은 벤더 자유. HotSpot은 Java 8([JEP 122](https://openjdk.org/jeps/122))부터 PermGen을 제거하고 이를 네이티브 메모리 영역인 Metaspace로 구현했다.
- PermGen→Metaspace 전환 시 클래스 메타데이터는 네이티브 메모리(Metaspace)로, interned String과 클래스 static 변수 값은 **Heap**으로 이동했다. 즉 "static 변수는 메타스페이스에 있다"는 통념은 부정확 — 값 자체는 Heap 객체이고, Metaspace에는 클래스 구조·메서드 코드·런타임 상수 풀이 위치한다.
- 스택을 스레드별로 분리하는 이유: 스택은 메서드 호출 깊이·지역변수를 담는데, 공유하면 스레드 간 호출 스택이 서로 덮어써져 동시성 보장이 불가능해진다.
