# `@ElementCollection` vs `@Embedded` vs `@OneToMany`

JPA에서 값 타입/엔티티를 어떻게 매핑하느냐에 따라 설계와 성능, 유지보수성이 크게 달라진다.  
여기서는 세 가지 대표 매핑인 `@ElementCollection`, `@Embedded`, `@OneToMany`를 비교한다.

---

## 1. 공통 개념 정리

- **엔티티(Entity)**  
  - 식별자(`@Id`)를 가지며 독립적인 생명 주기를 갖는 객체  
  - 별도 테이블에 매핑되고, 영속성 컨텍스트에서 관리된다.

- **값 타입(Value Type)**  
  - 식별자가 없고, 소유자 엔티티에 종속적인 객체  
  - 생명 주기가 소유자에 완전히 의존하며, 공유/참조하지 않는 것이 원칙

`@OneToMany`는 주로 엔티티 컬렉션을,  
`@ElementCollection` / `@Embedded`는 값 타입을 다룰 때 사용한다고 이해하면 된다.

---

## 2. `@Embedded`

### 특징

- **단일 값 타입(컴포넌트)**을 엔티티 필드에 **포함**시키는 방식
- **별도 테이블을 만들지 않고**, 같은 테이블에 컬럼을 펼쳐서 매핑
- 값 타입 클래스에 `@Embeddable`을, 엔티티 필드에 `@Embedded`를 사용

### 장점

- 테이블 수 증가 없이 **모델을 잘게 나눌 수 있어 응집도 향상**
- 재사용 가능 (여러 엔티티에서 같은 값 타입을 사용)
- 성능상 유리 (조인이 필요 없음)

### 단점 / 제약

- 컬렉션을 표현할 수 없음 (복수 값은 `@ElementCollection` 또는 관계 매핑 필요)
- 동일 테이블 컬럼 수 증가

### 언제 쓰는가?

- 주소, 금액, 기간, 좌표 등 **여러 필드를 하나의 개념으로 묶고 싶을 때**
- 복잡한 도메인 값을 모델링하면서도 테이블은 단순하게 유지하고 싶을 때

---

## 3. `@ElementCollection`

### 특징

- **값 타입 컬렉션**을 매핑할 때 사용 (`List`, `Set` 등)
- 별도의 **컬렉션 테이블**이 생성되며, 보통 소유자 엔티티의 PK를 FK로 가진다.
- 컬렉션에 담기는 타입은 `@Embeddable` 값 타입이거나, `String`/`Integer` 같은 기본 값 타입.
- 식별자가 없으므로 컬렉션의 각 원소는 개별적으로 식별되지 않는다.

### 장점

- 값 타입 컬렉션을 간단하게 표현 (추가적인 엔티티/ID 없이 사용 가능)
- 소유자에 종속적인 부가 정보(태그, 속성 값 목록 등)를 표현하기 좋다.

### 단점 / 주의점

- **식별자가 없기 때문에 변경 감지(update)가 어렵고, 보통 통째로 삭제 후 다시 insert**  
  → 변경이 잦고 크기가 큰 컬렉션에는 부적합
- 다른 엔티티에서 참조할 수 없음 (공유 금지)
- 컬렉션 테이블의 로딩 전략(LAZY/즉시 로딩) 및 N+1 문제에 유의해야 함

### 언제 쓰는가?

- 소유자 엔티티에 종속적인 단순 값 목록
  - 예: 사용자 태그 목록, 속성 키-값 목록, 취미 리스트 등
- **공유/참조/독립 생명주기가 필요 없는 컬렉션**일 때

---

## 4. `@OneToMany`

### 특징

- 엔티티와 **다른 엔티티의 컬렉션** 사이의 1:N 관계
- 관계의 반대편에 `@ManyToOne`이 있는 양방향 매핑이 일반적
- 컬렉션 쪽 엔티티는 **자신만의 ID를 가진 독립적인 엔티티**

### 장점

- 각 원소가 엔티티이므로 **식별자 기반 변경/삭제/조회가 가능**
- 여러 곳에서 **공유, 참조**할 수 있음
- 비즈니스 규칙/행위를 엔티티에 캡슐화할 수 있음

### 단점 / 복잡성

- 엔티티 수, 테이블 수 증가
- 연관관계 관리(연관관계 편의 메서드, cascade, orphanRemoval 등)를 신경 써야 함
- 조인과 지연 로딩, N+1 문제 등 성능 고려 필요

### 언제 쓰는가?

- 컬렉션의 각 원소가 **독립된 개체**로서 의미가 있을 때
  - 예: 주문(Parent) – 주문상품(Child), 게시글 – 댓글, 카테고리 – 하위 카테고리 등
- 별도로 조회/수정/삭제해야 하는 데이터일 때

---

## 5. 세 가지 매핑 비교

### 관점별 비교표

| 항목 | `@Embedded` | `@ElementCollection` | `@OneToMany` |
|------|-------------|----------------------|--------------|
| 대상 | 단일 값 타입 | 값 타입 컬렉션 | 엔티티 컬렉션 |
| 테이블 구조 | 같은 테이블 | 별도 컬렉션 테이블 | 자식 엔티티 테이블 |
| 식별자 | 없음 | 없음 | 있음 (`@Id`) |
| 생명주기 | 소유자에 종속 | 소유자에 종속 | 독립적(대부분) |
| 공유/참조 | X (원칙적으로) | X | O |
| 변경 감지 | 소유자 변경 감지에 포함 | 보통 통째 삭제/재생성 | 각 엔티티 단위로 가능 |
| 모델 복잡도 | 낮음 | 중간 | 높음 |
| 성능 | 조인 없음, 단순 | 조인 1번 추가 | 조인 및 연관관계 고려 필요 |

---

## 6. 설계 시 선택 기준

1. **이 데이터가 독립적인 엔티티인가?**
   - 예: 주문상품, 댓글처럼 독립적으로 의미가 있고, 별도 조회/수정이 필요하다 → `@OneToMany`(엔티티)

2. **소유자에 완전히 종속되는 값인가?**
   - 예: 주소, 이름, 기간, 좌표, 옵션 값 등 → 값 타입으로 모델링 (`@Embedded` 또는 `@ElementCollection`)

3. **단일 값 vs 컬렉션인가?**
   - 단일 값 → `@Embedded`
   - 컬렉션 → `@ElementCollection` (값 타입) 또는 `@OneToMany` (엔티티)

4. **변경 패턴과 크기는 어떤가?**
   - 자주 변하고, 개별 항목 단위로 수정/추적해야 한다 → `@OneToMany`
   - 비교적 작고, 통째로 갈아끼우는 형태의 변경이 많다 → `@ElementCollection`도 고려 가능

5. **성능과 단순성의 균형**
   - 필요 이상으로 엔티티/연관관계를 늘리면 복잡도만 증가할 수 있음  
   - 반대로, 엔티티로 다뤄야 할 대상을 값 타입으로 눌러 담으면 확장성과 표현력이 떨어짐

---

## 7. 요약

- `@Embedded`는 **단일 값 타입**을 같은 테이블에 포함시키는 가장 단순한 방식이다.
- `@ElementCollection`은 **값 타입 컬렉션**을 위한 편의 기능이지만, 식별자 부재로 인해 변경/성능 면에서 주의가 필요하다.
- `@OneToMany`는 **엔티티 컬렉션**으로써 가장 표현력이 크지만, 설계와 성능 고려가 가장 복잡하다.

결론적으로,  
**“독립 엔티티냐, 값 타입이냐” → “단일 값이냐, 컬렉션이냐”**  
이 두 질문에 답하면서 세 가지 매핑을 선택하는 것이 좋다.
